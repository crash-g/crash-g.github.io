1 Computer science

1.1 Algorithms   algorithm

1.1.1 Substring   string substring search

Find all occurrences of one string into another. This can be solved in linear time.

  1. Boyer-Moore   boyer moore

    The running time is O(n+m) in the revised version (the original version was not linear). The main ideas are to test matches proceeding backwards and to skip sections of the strings that cannot lead to a match. It works better when pattern and text to search resemble "natural text".

  2. Knuth Morris Pratt   knuth morris pratt

    The running time is O(n+m). The main idea is that when a mismatch occurs, the word itself embodies sufficient information to determine where the next match could begin, thus bypassing re-examination of previously matched characters. It works better when the pattern is a small set.

1.1.2 Hamming weight   hamming

Count the number of bits set to one in a positive 32-bit integer.

  1. Brian Kernighan's algorithm (medium)
    unsigned int countSetBits(int n)
{
    unsigned int count = 0;
    while (n)
    {
      n &= (n-1) ;
      /* The least significant bit which was set to 1 is set to zero by this line. */
      count++;
    }
    return count;
}

    Complexity is O(log n) or O(s) where s is the number of bits set to 1.

  2. Lookup table (easy and efficient, but expensive)

    Basically precalculate all results and put them in an array (complexity is O(1)). To keep the size of the array manageable, this could be done only on chunks of a fixed size (say 8 bits) and then the final result is the sum of all partial results.

  3. SWAR (probably best solution, but not very easy to read)
    /* unsigned int must be 32 bits */
int SWAR(unsigned int i)
{
    i = i - ((i >> 1) & 0x55555555);
    i = (i & 0x33333333) + ((i >> 2) & 0x33333333);
    return (((i + (i >> 4)) & 0x0F0F0F0F) * 0x01010101) >> 24;
}

    The idea is that the total is calculated "in parallel" on all bits at the same time, starting with two-bits chunks, then 4-bits, then all. Full explanation is here. Complexity is O(1).

1.1.3 Median   median

The median is the middle value separating the lower half from the higher half of a data sample. Given an array, the easiest way to find the median is sorting it and taking the middle value or the average of the two middle values.

  1. Quickselect   quickselect quicksort

    One of the best algorithms to find the median in practice. In its general form, it finds the k-th smallest element of an array. In short, it is a modified version of quicksort, where only one recursive call is made at each time: this is possible because at each recursive step of quicksort, the pivot already is in its final position, so it is known in which of the two subarrays the target element lies. The worst-case complexity is O(n^2), but the expected complexity is O(n).

    1. Median of medians

      This approach can be used to bring the worst-case complexity down to O(n), but it is often not useful in practice. The idea is to choose the pivot in a smart way, that is, as the median of the array of medians of the n/5 subarrays of length 5. This works because this pivot ensures that at most 70% of the elements are in the subarray over which quickselect recurses. The same pivot-selection strategy can be used to bring the worst-case complexity of quicksort down to O(n\log n).

1.1.4 Sorting   sort

  1. Bucket sort   bucket

    It is possible to assign each element to a different "bucket" and then sort each bucket separately. If the input to sort is uniformly distributed, buckets will be small and thus they can be sorted in O(1) time. The total running time is O(n).

  2. Counting sort   counting

    Just count the occurrences of each element and store them in an array (the index being the element whose occurrences have been counted). Running time is O(n), but this solution requires a large array to count the occurrences, which may not be feasible.

  3. Heap sort   heap

    Use a heap to sort. Most heap operations take O(\log n) time, so the total running time is O(n\log n).

  4. Insertion sort   insertion

    Simple sorting algorithm, easy to implement but with quadratic running time. Advantages are:

    • Efficient on small data sets.
    • Adaptive (O(kn) time complexity if no element is more than k places away from its final position).
    • Stable (does not change the order of elements with equal keys).
    • In-place (O(1) space-complexity).
    • Online.

    It processes an array one element at a time, comparing each element with the preceding ones and placing it in the correct position.

  5. Merge sort   merge

    Divide-and-conquer algorithm. Divide into n arrays, merge each pair, then merge pairs of pairs and so on. Merging sorted arrays is fast, so the total running time is O(n\log n).

1.1.5 Type inference   type

  1. Hindley-Milner   hindley milner

    Smart algorithm used for inferring value types based on use. It is implemented by languages like Haskell, but not by languages like Java.

1.2.3 Maven   maven

Software project management and comprehension tool whose goal is to manage project build, reporting and documentation from a central point.

  1. Build lifecycle   build lifecycle

    Central concept in Maven, a build lifecycle is a clearly defined process related to the building and distribution of a project. There are three built-in build lifecycles: default (handles project deployment), clean (handles project cleaning) and site (handles the creation of a project's site documentation).

    1. Phase

      A phase is a step in a build lifecycle, which is an ordered sequence of phases. When a phase is given (like in the command mvn package), Maven will execute every phase in the sequence up to and including the one defined. Phases, actually, are mapped to underlying goals, but which goals depends upon the packaging of the project (for instance, the package phase will run different goals if the project is a JAR or a WAR).

      1. Default lifecycle phases

        The default lifecycle includes the following phases:

        • validate
        • compile
        • test
        • package
        • verify
        • install
        • deploy
      2. Mapping goals to phases

        The most common way is just to define a packaging for the project. For additional control, plugins can be configured through the <plugin> tag. When adding a plugin, it should be specified which ones of its goals must be executed and in which phase, but often plugins have a standard setting. Goals from a plugin are added to one or more of the phases and are generally executed after the phase's goals.

        1. Execution

          The <executions> tag can be used to modify the execution order, specify which goals to execute and in which phase, assign an ID (useful for inheritance)…

  2. Plugin   plugin

    Maven is, at its heart, a plugin execution framework: all work is done by plugins. Some of them are core plugins and the rest are provided by third-parties. A plugin can be thought of as a collection of goals with a general common purpose. When using a command like mvn archetype:generate ... the plugin is archetype and the goal is generate. Plugins are automatically downloaded when needed (by default, the latest version is used).

    1. Build plugins

      Plugins that are executed during the build. They should be placed inside the <build> tag.

    2. Reporting plugins

      Plugins that are executed during the site generation. They should be placed inside the <reporting> tag.

  3. Profile   profile

    Unfortunately, builds are not always portable. Maven uses profiles to specify system-specific or user-specific settings. A profile element contains both an optional activation (which is the profile trigger) and the set of changes to be made to the POM.

  4. Project Object Model (POM)   pom

    This is the fundamental unit of work in Maven. It is an XML file that contains information and configuration details about a project (note that in Maven everything revolves around the notion of project). When executing a task or goal, Maven looks for the POM in the current directory.

    1. Aggregation   aggregation

      This is a sort of reversed inheritance. It is used to specify modules from the parent POM. If a Maven command is executed against the parent project, it will be executed to the parent's modules as well. In addition, some commands behave differently over the modules. To use aggregation, the parent POM packaging must be pom and modules must be specified inside the <modules> tag. The relative path of modules must be specified.

    2. Inheritance   inheritance

      A POM always inherits from a parent POM (which may be the super POM). The parent POM may be set using the tag <parent>. Inherited elements are

      • Dependencies
      • Developers and contributors
      • Plugin lists
      • Plugin executions with matching IDs
      • Plugin configuration
      • Resources

      Inherited elements are merged with new elements (obviously new ones take precedence).

      1. Dependencies   dependency

        Dependencies specified in the <dependencies> element of the parent are always considered dependencies of all its children. Dependencies specified in <dependencyManagement> are only included if specified in the <dependencies> element of the child too. This is useful because the parent could specify the scope and/or version and there is no need to repeat this information in every child.

      2. relativePath

        When inheriting from a POM which is not associated to a project in the local repository and which is not in the parent directory relative to the child POM, the relative path must be explicitly given using the tag <relativePath> inside the <parent> tag.

    3. Minimal POM

      Example of a minimal POM: all the properties are required. 

      <project>
  <modelVersion>4.0.0</modelVersion>
  <groupId>com.mycompany.app</groupId>
  <artifactId>my-app</artifactId>
  <version>1</version>
</project>

      Note that if explicitly extending a parent POM, some of these properties can be skipped (for instance, <version>).

    4. Super POM

      This is Maven's default POM. All POMs directly extend the super POM unless a parent is explicitly set.

  5. Resource files   resource

    Normally, these files are contained in resources folders (see Standard Directory Layout).

    1. Filter resources

      Setting filtering to true, Maven scans the resources and dynamically replaces the values of placeholders which depend on build time information. Placeholders look like ${property.placeholder}. If the name starts with project, they refer to values defined in the pom.xml as, for instance, ${project.name} (also default values not explicitly defined can be used). If the name starts with settings they refer to the settings.xml. External files can be used if registered in the pom.xml; otherwise, the properties section of the pom can be used too.

  6. SNAPSHOT   snapshot

    The development version is always labelled with SNAPSHOT, as in 1.0-SNAPSHOT. When a project with version x.y-SNAPSHOT is released, its version becomes x.y and the new development version becomes x.(y+1)-SNAPSHOT.

  7. Standard Directory Layout

    src contains main and test, both with a java folder inside and then the packages. The compiled classes end up in target. Inside main there can also be the folder resources, which will be added to target as is. The same holds for test. By default, Maven also packages the pom and a properties file, useful to retrieve build info from the application.

  8. Troubleshooting   troubleshooting
    1. Running on a JRE rather than a JDK

      This is a problem with the Java compiler. It may depend on many things, but in one case on Linux was an incomplete installation of the JDK. It was solved running sudo apt-get install openjdk-8-jdk openjdk-8-doc openjdk-8-jre.

1.3 Business Intelligence System (BI)   bi

A system designed to extract meaningful data from a data source. The main components of a generic BI system are the ETL, DW, and analysis and reporting.

1.3.1 Extraction, Transformation, Loading (ETL)   etl

Since a data source can be of any kind (and often there are multiple different ones), and since a data source is designed and (often) actively used to deliver a specific service, it is better to extract the relevant data and load it on a separate system. The transformation part is the process of normalizing data representation in order to store it in a uniform and meaningful form. The process may also require a staging area as a temporary data storage in order to stress the data sources as little as possible.

1.3.2 Data warehouse (DW)   dw

This is just a database used to store the extracted information. Nonetheless, due to the specific requirements of this DB, it is usually designed in a different way (thus the different name). One of the possible design schools is the Kimball one, which requires data marts (separate sections for different data domains) and dimensional modeling (non-relational design with facts tables to store measures and dimension tables to store accessory data).

1.3.3 Online analytical processing (OLAP)   olap

Category of database processing used for multidimensional analysis. The use of measures and dimensions translates almost directly to the concept of a cube, that is, a cell in a multidimensional space which contains aggregated measures. This approach enables swift extraction of data to construct reports. One of the requirements is the possibility to change the granularity of the dimensions in order to change the granularity of the results.

1.4 Computer architecture

1.4.1 Difference between stack and heap   stack heap

The main difference is the allocation/deallocation algorithm. The stack implements a stack data structure, which makes it trivial to allocate and deallocate memory on top of it (O(1) time complexity). This makes it ideal for objects whose lifecycle follows strict rules, such as function arguments, but not for objects which are passed around. The heap, on the other hand, with O(\log n) running time for both allocation and deallocation, is ideal for these more "dynamic" objects. Also, the stack maximum size is determined when the process starts, while the heap size is on-demand (but I think this is not a hard requirement), so the heap is more suited for large objects. Note that both are in the computer RAM. Usually there is one stack per thread and only one heap for the whole process.

1.4.2 Difference between thread and process

Basically, a process is made of one or more threads.

  1. Process   process

    A process provides the resources needed to execute a program. It has a virtual address space, executable code, security context, priority class, and some other stuff. Every process has its own separate memory space: so, some given memory location, say 0x8000, will be mapped to different physical memory address by different processes.

  2. Thread   thread lwp

    A thread, also called lightweight process (LWP) in some systems, is an entity within a process that can be scheduled for execution. All threads share a virtual address space (that is, two threads accessing a memory location will in fact access the same physical memory address) and system resources. In addition, a thread maintains exception handlers, thread local storage, unique thread identifier, a set of structures used by the system to suspend and later resume the execution, and other things. In modern operating systems, threads are managed by the OS itself.

    1. Scheduling   scheduling

      Across platforms, thread scheduling tends to be based on the following criteria:

      • a priority, or in fact multiple priority settings,
      • a quantum, or number of allocated timeslices of CPU (it basically determines the amount of CPU time a thread is allotted before it is forced to yield the CPU to another thread with the same or lower priority),
      • a state, notably running or waiting,
      • various metrics about the behavior of threads, such as recent CPU usage or time since last run.

      The idea is that a thread of higher priority will preempt a thread of lower priority. Otherwise, threads of equal priorities will take turns at getting an allocated slice of CPU.

      1. Quantum   quantum

        A quantum is generally defined in terms of clock ticks, which are fixed units of time which depend on the OS. Calculations in this context are often approximated in order to favor speed.

      2. Switching   switch

        At key moments, the thread scheduler considers whether to switch the thread which is currently running. Usually these moments are:

        • periodically, via an interrupt routine, the scheduler will check if the running thread has reached the end of its allotted quantum,
        • at any time, if the running thread ceases to be runnable (e.g. by needing to wait for a resource),
        • when some other attribute of the thread changes (e.g. its priority).

        At these decision points, the scheduler's job is to decide, of all the runnable threads, which are the most appropriate to actually be running. Potentially it is a complex task, but heuristics are used. Basically, if a thread with higher priority is found, the running thread is preempted (but allowed to keep the remaining quantum). If the allotted quantum has been used, the thread is preempted and put at the end of a queue of same priority threads. If nothing better is found, the thread could immediately continue running. To complicate things, there are also priority boostings and penalties, depending on the OS.

        1. Overhead

          Switching running threads has a cost, mainly because there is an interruption of the instruction pipeline and the cache is often not relevant anymore. If switching between different processes, the address-to-memory mapping must be updated too.

      3. State   state
        • Runnable: ready to consume CPU,
        • waiting: cannot continue as it is waiting for a resource such as a lock or I/O,
        • terminated: thread's code finished but not all of the resources have been cleaned,
        • new: the thread has just been created but not all resources have.

1.4.3 Master boot record (MBR)   mbr

First sector of an hard disk, with 0 as the offset. When a computer boots, the BIOS looks for configured primary bootable devices (hard disk, USB keys and so on) and executes the initial bootstrap program contained in the MBR. Since 2009 there are hard disks with sector size of 4096 bytes, but usually the sector size is still 512 bytes. The legacy MBR partition table supports at most 4 partitions and occupies 64 bytes; in addition there are 4 bytes for disk signature and 6 bytes for the timestamp, which leaves 434 bytes for the bootstrap program. Often this space is inadequate, so boot loaders only fit part of themselves in the MBR and from there call codes in different parts of the disk.

1.4.4 Random Access Memory (RAM)   ram

A type of computer data storage that can be accessed in a random manner, instead of an arranged order. Currently, RAM storage has fast access but is volatile (which means that it requires energy to keep information).

1.4.5 Read-only Memory (ROM)   rom

A type of computer data storage that is read-only. Normally, a ROM is also a RAM, in the sense that it permits random access. Nevertheless, it is non-volatile.

1.4.6 Volume boot record (VBR)

It is a type of boot sector, corresponding to the first sector of a partition (or of the whole disk if the disk is non-partitioned). The code in a VBR is invoked directly by the firmware or indirectly by code in the MBR or by a boot manager (chain loading).

1.5 Database   database db

1.5.1 Database relazionale (relational database)

Database in cui le tabelle corrispondono a entità differenti e ogni riga è identificata da una chiave univoca che può essere usata per riferirsi a quella riga in altre tabelle. Esempio principale: SQL.

1.5.2 Modello concettuale (conceptual model)

Modello di ragionamento che studia un problema dal punto di vista di concetti (entities) e relazioni tra essi, precedentemente all'implementazione. L'obiettivo primario è chiarire i punti ambigui e presentare un modello, spesso visivo, strutturato e, per quanto possibile, indipendente da design e altre considerazoni specifiche che verrano poi adottate nell'implementazione. Esistono alcuni metodi standard di produrre un modello concettuale (non tutti necessariamente applicabili in ogni contesto), di cui uno è il MODELLO ENTITÀ-RELAZIONE.

1.5.3 Modello entità-relazione (entity-relationship model)

Modello per rappresentare dati in termini di entità e relazioni tra esse, in modo che si presta ad essere poi implementato in un DATABASE RELAZIONALE. In aggiunta, sia le entità che le relazioni possono avere zero o più attributi. Ci si concentra sui rapporti tra entità diverse tralasciando la maniera in cui vengono poi utilizzate. Esistono solitamente vari livelli standard di granularità nell'analisi di un problema. Visualmente, il modello viene rappresentato con diagrammi in cui le entità sono rettangoli, le relazioni sono rombi e gli attributi sono ovali. Chen (creatore del modello) ha proposto la seguente approssimativa trasposizione da linguaggio a modello:

Language Model
Common noun Entity type
Proper noun Entity
Transitive verb Relationship type
Intransitive verb Attribute type
Adjective Attribute for entity
Adverb Attribute for relationship

1.5.4 MultiDimensional eXpressions (MDX)   mdx bi

It is a query language for OLAP databases developed by Microsoft.

  1. Keywords
    1. Cube

      Subsets of data from an OLAP store, organized into a multidimensional structure. This structure is motivated by the necessity of having rapid and uniform response times to complex queries.

    2. Measure

      Numerical value that is summarized for analysis (such as price, quantity sold…).

    3. Dimension

      Categorical description by which measures are separated for analysis (such as time, space…). Note that the collection of measures form a special dimension, called Measures. A dimension can be private if it is used by only one cube, otherwise it is shared.

    4. Hiearchy

      Each cube dimension can contain a hierarchy of levels, corresponding to a gradually narrowing scope of analysis (for instance, state -> region -> city). Multiple independent hierarchies may exist in the same dimension.

    5. Level

      A level in a hierarchy.

    6. Member

      It is an item in a dimension or a hierarchy. A calculated member, on the other hand, is a member whose value is calculated at run time: only the definition of such members is stored.

    7. Tuple

      Ordered collection of one or more members from different dimensions. It is specified as (member1, member2, ...).

    8. Set

      Ordered collection of tuples with the same dimensionality or hierarchality. It is specified as {tuple1, tuple2, ...}.

  2. Functions
    1. MEMBERS

      Returns all the members in the specified level (in the extreme case of a hierarchy, it returns all the members of that hierarchy).

      1. Usage

        [dim.hier].[lv1].[lv2].MEMBERS returns all the members in lv2.

    2. CHILDREN

      Returns all the children of the specified member. As a special case, if applied on a hierarchy, it returns all the members of the first level.

      1. Usage

        [dim.hier].[lv1].[lv2_member].CHILDREN returns all the children of lv2_member.

    3. PARENT

      Returns the parent of the specified member.

      1. Usage

        [dim.hier].[lv1].[lv2_member].PARENT returns the parent of lv2_member.

    4. DESCENDANTS(member, level[, flag])

      Returns all the members of level which are descendants of member. Flag can be SELF, BEFORE, AFTER, BEFORE_AND_AFTER and is used to decide which levels should additionally be considered, if any.

      1. Usage

        DESCENDANTS([dim.hier].[lv1].[lv2_member], [lv4]) returns all the members of lv4 which are descendants of lv2_member.

    5. NON EMPTY

      If put in front of an axis specification, it filters rows where every cell is blank.

    6. NONEMPTYCROSSJOIN(set1, set2)

      Returns a set which is the cartesian product of set1 and set2, with blank rows filtered out.

    7. IIF(condition, value1, value2)

      It is like the ternary operator in Java.

    8. HIERARCHIZE(set)

      Applies the default order on a set.

    9. ORDER(set, measure[, flag])

      Orders a set by the given measure. Flag can be BASC, BDESC, ASC, DESC. The B stands for breaking: without, the level hierarchy is not broken by the ordering.

  3. Basic query structure

    [WITH (MEMBER|SET) name AS 'expression'] SELECT axis specification ON COLUMNS, axis specification ON ROWS FROM cubename [WHERE slicerspecification]

    1. Axis specification

      It specifies the members that are to be included in a given axis. It is given as a set.

    2. WITH

      The WITH clause is an optional definition of calculated members or named sets. In the former case, it allows to define a formula and treat it as a new member of the specified parent: the MEMBER keyword is used and the name must be in the form parent_name.name. In the latter case, it allows to create a custom set: the SET keyword is used.

    3. WHERE

      The WHERE clause is an optional restriction which defines the slice of the cube to be viewed. It is given as a tuple. It is completely different from an SQL WHERE: note, for instance, that an MDX WHERE influences the value of the results, not their quantity, whereas an SQL WHERE influences the quantity of results but not their value. Note that in a WHERE clause it is NOT possible to use the same hierarchies used in an axis specification.

  4. Troubleshooting

    Brackets must be used if the name within contains spaces. When specifying the hierarchy of a dimension which only contains one hierarchy, it is enough to write [dim] instead of [dim.hier]. The where condition effectively shrinks the cube, filtering out part of the value of the measures. This influences every other part of the query, including calculated members.

    1. Members and aggregation

      Normally aggregation is performed over all data associated to a given member: that is, if a month is fixed, say January, measures are aggregated over all Januarys, regardless of the year. Nonetheless, if selecting January as a child member of a year, say [TimeDimension.TimeHierarchy].[Year].[2011].[1], aggregation should take the parent-child relationship into account and aggregate over the correct subset. With Pentaho, this is not necessarily the case. It appears the behaviour depends on the value of the uniqueMembers property (which is found at the level level). When last tried, everything worked when the property was set to false.

1.5.5 Transaction   transaction

Unit of work within a database management system. A transaction must be atomic (it must be an indivisible and irreducible series of database operations), consistent (it must leave the database in a coherent state), isolated (it must not influence or be influenced by other transactions) and durable (even a system crash should not undo the effect of a successful transaction).

  1. Container-managed

    In an enterprise bean with container-managed transaction demarcation, the EJB container sets the boundaries of the transactions (typically, the start and end of methods). Methods should not attempt to explicitly start or end transactions in this case: the class and method tag @TransactionAttribute can be used to define the behaviour of EJB methods (see here). Rollback happens in case of exception or if explicitly required by calling setRollbackOnly() on the session context.

  2. XA and non-XA

    An XA transaction is a "global" transaction (or distributed transaction) that may span multiple resources (such as multiple databases). A non-XA transaction always involves just one resource. The former type requires a coordinating transaction manager that implements a Two Phase Commit (2PC) protocol (first apply changes everywhere, then commit everywhere). The single resources should support this protocol too and, in particular, should implement a Resource Manager. Transaction manager and resource managers together implement the observer pattern. In case of a JavaEE application, the app server acts as a transaction manager. Note that most stuff in the world is non-XA.

    1. Datasource

      An XA datasource is a datasource that can participate in an XA transaction. A non-XA one cannot (sort of, there are exceptions).

1.5.6 View and Materialized View

A view is only logical, it is produced by a SELECT statement which is run every time the view is queried. So its performance depends on the performance of the underlying select. A materialized view is a view which is persisted on the database as a real table. It is not refreshed every time it is queried (different refresh schemes exist, including every time the underlying data changes). Since it is a table, it can be indexed. In addition, Oracle may dynamically change queries in order to use it if it is more efficient. See query rewrite.

  1. Git   git

      2. Add to .gitignore   gitignore

      3. Find file in history   find search

        git log --all --full-history -- **/thefile.* finds the hash of the commits that contain thefile, then git show <SHA> -- **/thefile.* shows it on screen.

      4. Ignore files   ignore

        If the file is untracked, it is enough to add it to .gitignore. Otherwise, run git update-index --assume-unchanged file_to_ignore. To see which files have been added to the assume-unchanged list, run git ls-files -v | grep -e "^[hsmrck]". To track again, git update-index --no-assume-unchanged file_previously_ignored. Note that in any case when pulling from a remote repository the files will still be overridden!

      5. Remove files from source control   remove

        git rm --cached file_to_remove adds a file deletion to the stage, without physically deleting the file in the local repository (note that it will be deleted in the remote one though). See also 1.7.4.1.2.16.

    2. Basic commands

      Most of these commands can be followed by a file name (e.g. git diff --cached -- myFile.txt) to restrict the output to a single file. The -- is probably optional, it tells git that the list of options is finished and the list of files is starting.

      1. branch   branch

        git branch lists all branches. git branch branchname creates a branch named branchname. With -d option it removes an existing branch. git checkout branchname changes the context to branch branchname. With -b option it immediately switches to the new branch.

      2. checkout   checkout

        On the commit-level, it is a way to move HEAD to a different commit (and update the working directory accordingly). Contrary to 1.7.4.1.2.13 nothing is lost with this command (because the HEAD is moved instead of the tip of a branch), but if more work must be done on old commits then a new branch should be created first. On the file-level, it is similar to reset but it updates the working directory instead of the stage. Therefore, it can be used to discard unstaged changes. If the file is then staged and committed, all changes are overwritten. git checkout - brings back to the latest commit. git checkout <deleting_commit>^ -- <file_path> is used to undelete a file. <deleting_commit> is the identifier of the commit that deleted the file. git checkout --orphan <new_branch> creates a new branch without the old commit history.

      3. commit   commit

        -a option is used to automatically stage and commit all (and only) modified files.

      4. config   config

        This command can be used to write configuration. There are three levels of configuration, system, global and local. Note that they are applied in this order (so they overwrite each other).

        1. get   get

          The --get option is used to get the value for a given key. For example, git config --global --get http.proxy.

        2. alias   alias

          It is possible to create aliases for commands. For example, git config --global alias.st status, which makes git status and git st equivalent.

        3. global   global

          This options are written in ~/.gitconfig.

          1. Editor   editor

            git config --global core.editor emacs configures emacs as the default editor.

          2. Personal data   name email

            git config --global user.name John Doe configures John Doe as default user name. git config --global user.email johndoe@example.com configures default email.

          3. Proxy   proxy

            git config --global http.proxy http://proxyuser:proxypwd@proxy.server.com:8080 sets an http proxy.

      5. diff   diff

        git diff shows the not yet staged differences from last commit. git diff --cached shows the staged differences from last commit. git diff HEAD shows all differences from last commit. With HEAD@{n} we select the n-th previous commit instead. With option --name-only shows only the names of the files that are changed.

      6. log   log

        git log shows a history of commits. If a list of files is specified, the output is limited to these files. The --oneline flag gives shorter output. The --decorate flag shows all the references in addition. The --stat flag shows the modified files for every commit and shows statistics for each file (insertions and deletions). The -p flag shows the output of diff for each modified file. The --graph flag shows a graph of branches and merges. The --pretty=format: <string> is used to fully format the output. <string> is constructed using fixed placeholders (see documentation). The --after and --before options filter the output by date. It can be used as –after="year-mm-dd" or –before="yesterday. The --author option filters by author and accepts a regexp as argument. Same thing with --grep, except that this filters by commit message (with the -i parameter it also ignores case). The -S option is used to find when a specific string of text was added. Usage: git log -S"<string>". -G is the same but it accepts a regexp. The double-dot option specifies a range: git log master..feature shows all commits reachable from feature but not from master. The first reference should be interpreted as since and the second as until. The --no-merges flag excludes merge commits from the output. The --merges flag only shows merge commits.

        1. Advanced usage

          git log --format=format:%H $FILE | xargs -L 1 git blame $FILE -L $LINE,$LINE can be used to show the history of changes in a file $FILE for a specific range of lines. git log --pretty=format:'%C(yellow)%h %Cred%ad %Cblue%an%Cgreen%d %Creset%s' --date=short shows a nice log with authors' names, commit dates and pretty colors.

      7. merge   merge

        git merge branchname merges the branch called branchname to the current branch. In case of conflicts, after solving them manually it is also necessary to stage the files.

      8. merge-base   merge base

        git merge-base branch1 branch2 prints the hashsum of the first commit common to both branches.

      9. pull   pull

        git pull --rebase performs a 1.7.4.1.2.10 instead of a 1.7.4.1.2.7 when pulling from a remote branch.

      10. rebase   rebase

        This is an alternative command to 1.7.4.1.2.7, which can be used to avoid spurious merge commits. Its effect is to merge two branches moving one at the end of the other while preserving all commits that were made. Since this rewrites the history of the project, it can have catastrophic effects. git rebase master used from a branch feature moves the entire branch to begin on the tip of the master branch. With -i flag the rebase is interactive, which means that the commit history can be changed in the process.

        1. Golden rule

          Never use it on a public branch! The branch local version would become different from the public one and it would require a merge later on, which would contain a set of duplicate commits (the ones that always were in the public branch and the ones that the rebasing duplicated). Otherwise it is also possible to force things with git push --force, which should push the rebased master branch regardless of the differences, but this would change history for everybody and potentially create a lot of conflicts.

        2. Local cleanup

          git rebase -i HEAD~n is used to rewrite the history of the last n commits. It does not perform any merge operation, but if there is another branch one must be sure that common parts are not rewritten. 1.7.4.1.2.8 can be used to this purpose.

        3. Integrating approved feature

          Instead of merging master to feature and then pushing, it is possible to rebase and then pushing, which will still produce a merge commit but it will just be a fast-forward, thus keeping the history clean. In order not to mess things up, a good way of doing it is: 

          git checkout feature
git checkout -b temporary-branch
git rebase -i master
[Clean up the history]
git checkout master
git merge temporary-branch

        git reflog [<ref>] shows the log history of the provided reference (HEAD by default). This will not show history which has been erased because it is too old.

      12. remote   remote

        git remote show origin shows info about the origin repository. git remote set-url origin NEW_URL changes the origin repository URL to NEW_URL.

      13. reset   reset

        On the commit-level, it is a way to move the tip of a branch to a different commit, sending to garbage collection anything that is older. There are three modes which have an incrementally broader effect: --soft (which moves the pointer and does nothing else), --mixed (the default mode, which also updates the staged snapshot, unstaging what it should) and --hard (which also updates the working directory, deleting what it should). On the file-level, it is a way to update the staged snapshot of a file. There are no modes in this case, and the working directory is never updated. git reset HEAD is used to unstage files. git reset --soft HEAD~ is used to undo a commit (it leaves files on stage, so can be used to do more work before committing again). git reset --hard HEAD rolls everything back to the last commit, deleting all changes since then. git reset HEAD~ <file_name> fetches the version of <file_name> referenced by HEAD's parent and stages it for commit.

      14. rev-parse   revparse

        git rev-parse [<ref>] shows the SHA-1 signature of <ref>. It is a plumbing tool not intended for normal usage.

      15. revert   revert

        This is a tool to undo a commit by creating new commits. It should be used instead of reset on public branches, in order not to rewrite history. git revert HEAD~2 creates a new commit which corresponds to the parent of the parent of HEAD.

      16. rm   rm

        git rm file removes file from stage (and from disk). With --cached options it just removes them from stage.

      17. shortlog   shortlog

        git shortlog shows commits divided by author. The -n flag sorts the output by the number of commits.

      18. stash   stash

        git stash saves all staged files in a stack for later processing, then reverts everything back to last commit. With -u option it also includes all untracked changes. git stash list shows all stashed snapshots. git stash apply brings back the most recent stashed snapshot. git stash pop does the same but also removes it from the stack. stash@{n} can be added at the end to specify the n-th stashed snapshot. git stash drop drops the most recent stashed snapshot and git stash clear clears everything.

      19. tag   tag

        git tag lists all the tags. git tag <tag_name> [<ref>] creates a lightweight reference to <ref>, which is just a named version of that reference. git tag -a <tag_name> [-m "message"] creates an annotated reference to <ref>, which is a proper tag object.

    4. Theory   theory
      1. Reference

        An object is identified in Git through its SHA-1 hash. To make things easier Git uses references, which are simply files stored in .git/refs and containing the hash of a commit object. Git also has a special reference, HEAD, which is a symbolic reference pointing to the tip of the current branch.

      2. Reference log

        Git constantly keeps a reference log, that is a log of where the HEAD and branch references have been in the past few months. Note that branch changes are tracked in the history of HEAD.

      3. Repository

        A Git repository is a key-value collection. It contains blobs (a bunch of bytes, usually the binary representation of a file), tree objects (a bit like directories, they contain pointers to blobs or to other tree objects), commit objects (which contain a pointer to a single tree object and some meta data), tag objects (which contain a pointer to a single commit object and some meta data) and references (which are pointers to a single object, usually a commit or a tag object). Everything is contained in the .git directory in the root of the project.

      4. Revision selection

        There are many different ways to identify a commit:

        • The SHA-1 hash, or a short version of it (at least four characters, enough to make it unique).
        • The @{n} syntax: for instance, HEAD@{n} identifies the n-th previous commit HEAD pointed to. This cannot be used for commits older than reflog.
        • Temporal identifiers, as in master@{yesterday} or HEAD@{2.months.ago}.
        • Ancestry reference, either ^ (as in HEAD^2, which identifies the second parent commit of HEAD) or ~ (as in HEAD~3, which identifies the parent of the parent of the parent of HEAD).

1.11 Network   web network

1.11.1 Basic theory   theory

A computer network or data network is a digital telecommunications network which allows nodes to share resources. In computer networks, networked computing devices exchange data with each other using a data link.

  1. Difference between the Internet and the World Wide Web   internet www

    The Internet is a massive network of networks that connects millions of computers. Information over the Internet travels via a variety of languages known as protocols. The World Wide Web is a way of accessing information over the Internet, built upon the HTTP protocol. It is just a portion of the Internet, but one of the largest.

  2. Difference between the OSI and TCP/IP models   osi tcp ip

    The OSI model is a standard reference that describes how different software and hardware components involved in a network communication should divide responsibilities and interact. TCP and IP are protocols that describe how to create reliable channels of communication and how to route data over a network, respectively. They are core protocols of the Internet protocol suite. The model behind them is similar to the OSI model, but does not map cleanly on it. This is due to multiple reasons, one being that the OSI model was created ten years later. Another reason is that they were created to solve a specific set of problems, not to function as a general description of all network communications.

    A set of false assumptions usually made by beginners new to programming distributed applications.

    1. The network is reliable.

  4. Hostname / Domain name   hostname domainname

    Un hostname è il nome di un computer all'interno di una rete, mentre un domain name è la stringa identificativa di una porzione "indipendente" di internet. Il nome di un sito è formato da hostname (tipo www) e domain name (tipo wikipedia.org). Un computer può utilizzare svariati alias come hostname, ed essere quindi associato a più hostname.   5. Internet protocol suite   internet

    Conceptual model and set of protocols used on the Internet and similar networks.

    1. Hypertext Transfer Protocol (HTTP)   http

      HTTP is an application-layer protocol used by web servers and web browsers to communicate. It is based on a request (from the client) and a response (from the server). It uses TCP as transport protocol.

      1. HTTP persistent connection

        Opening a connection is costly (CPU and time, mainly due to DNS lookup and TCP handshake I think), so the same connection could be used for multiple requests/responses in a row. This is even more important for HTTPS connections. This is implemented with Connection: keep-alive header, but it is not standard. HTTP/2 actually provides this. Note that keeping a connection open is costly too (mainly memory, I think for the information associated to the connection), so this is one of the possible disadvantages of persistent connections.

      2. Postback

        It is an HTTP POST to the same page that the form is on.

      3. Request

        A request is formed by a first line containing type, webpage, parameters (separated by the webpage with a ? and by themselves with a &) and http version. After that there is the header which contains additional info and, possibly, a body. There are various types of requests, but the most common are GET and POST: the main difference is that parameters are sent in the first line for GET and in the body for POST.

      4. Response

        A response is formed (again) by a first line, header and body. The first line contains http version and status code (success, error, redirect…), the header contains info about the server and the body usually contains HTML, CSS, Javascript and/or binary data.

    2. Internet protocol (IP)   ip

      Communication protocol which is the main component in the Internet protocol suite. Its purpose is to regulate the delivery of packets based on the IP address provided. There are two main versions: IPv4 and IPv6.

    3. Transmission Control Protocol (TCP)   tcp ip http

      Transmission protocol which is part of the Internet protocol suite. It originated in the initial network implementation where it complemented the Internet Protocol (that is why it is commonly referred to as TCP/IP). It provides reliable, ordered and error-checked delivery of a stream of octets between applications running on hosts communicating by an IP network.

      1. Handshake

        To establish a connection, TCP uses a three-way handshake initiated by the client.

    4. User Datagram Protocol (UDP)   udp

      Transmission protocol which is part of the Internet protocol suite. It uses a simple connectionless transmission model without handshaking which gives no guarantee of delivery, ordering, or duplicate protection. It is useful for real-time applications which value speed more than data integrity, such as streaming and voice over IP (note that TCP will wait for delayed packages, blocking the transmission in the meanwhile). The package payload is called datagram (generic term which refers to variable length data sequences) and cannot be split upon multiple packages (such as in TCP).

  6. Local Area Network (LAN)   lan

    A network where all the devices are in the same subnet, that is, all IP addresses are locally reachable without the need of a gateway or router: only layers 1 and 2 are concerned during a data transmission on such a network. In practice, layer 3 gets mixed up with the rest too: applications usually talk using IP addresses, so the OS applies an Address Resolution Protocol (ARP) to retrieve the MAC address from the local IP address. Afterwards, the communication continues on layer 2, using the normal hub's or switch's capabilities.

    1. Address Resolution Protocol (ARP)   arp

      Example of operation:

      1. An application on IP 10.1.1.2 attempts to talk to an unknown local IP 10.1.1.1.
      2. A Broadcast Frame (destination MAC address ffff.ffff.ffff source MAC address 1234.5678.aaaa) is sent with an ARP message "Who is 10.1.1.1? Tell 10.1.1.2".
      3. Computer 10.1.1.1 receives the message and responds with a Frame (destination 1234.5678.aaaa, source 1234.5678.bbbb) with the ARP message "10.1.1.2 with MAC 1234.5678.aaaa; 10.1.1.1 has MAC 1234.5678.bbbb".
  7. Network Adapter   adapter

    A network adapter is the component of a computer's internal hardware that is used for communicating over a network with another computer.

    1. Virtual Network Adapter   virtual

      A virtual network adapter is a program (instead of a physical network adapter) that allows a computer to connect to a network.

  8. Networking devices
    1. Hub   hub

      A device used to connect multiple hosts on the same network. It operates at the physical layer level (layer 1): it has no memory to store data or to analyze it, it just reads it and forwards it to all its ports. For this reason, it is neither secure nor efficient. Note that, in general, a hub is the central part of a wheel where the spokes come together.

    2. Switch   switch

      Like a hub, it is a device used to connect multiple hosts on the same network. The difference is that it works at the data link layer level (layer 2): it is able to filter data depending on the destination (for instance, based on the MAC address) and it can also provide some error-checking or some other functions. These capabilities are usually implemented in hardware. Normally, it maintains a list of known addresses in order to perform the filtering. Some switches also understand layer 3.

    3. Bridge   bridge

      It is a device used to connect multiple hosts on different networks which use the same protocol. It is similar to a router, but it works at the data link layer lever (level 2) like switches do: simplifying, this means that it uses the MAC address instead of the IP address (but remember that MAC addresses are not the only possibility for layer 2).

      1. Difference with switches

        According to the IEEE standards, there is nearly no difference between switches and bridges: both can be used for the same purposes. In practice, there are technological differences (such as forwarding implemented in hardware for switches and software for bridges). Switches are apparently considered superior.

    4. Modem   modem

      It modulates and demodulates the signal between the digital data of a computer and the analogue signal of a telephone line. Its name stands for Modulator/Demodulator.

    5. Router   router

      It is a device used to route traffic from one network to another. It works at the network layer level (level 3), so it can connect networks which use different protocols. It uses routing tables and various strategies implemented in software in order to process and route packages. Routers can also be just software running on a computer, instead of separate devices.

  9. Open Systems Interconnection model (OSI model)   osi

    Conceptual model that characterizes and standardizes the communication functions of a telecommunication or computing system without regard to their underlying internal structure and technology. It partitions a communication system into abstraction layers. The message to exchange is initially produced by the topmost layer, then each layer in turns add a header, footer or both. Upon reception, the process is reversed, so starting from the bottom layer each layer in turns processes the message and strips the parts that are not meant for the higher layers.

    7. Application Data High-level APIs, including resource sharing and remote file access.
    6. Presentation Data Translation of data between a networking service and an application, including character encoding, data compression and encryption/decryption.
    5. Session Data Managing communication sessions, i.e. continuous exchange of information, back-and-forth.
    4. Transport Segment/Datagram Reliable transmission of data segments between points on a network, including segmentation, acknowledgement and multiplexing.

  10. Protocol stack   protocolstack networkstack

    It is an implementation of a computer networking protocol suite (such as the Internet protocol suite). It is also called a network stack. The term stack is motivated by the nature of the implementation, which in effect uses a stack of protocols. The idea is that the network can be very complicated, and it is impossible to enforce a unique protocol for all communication: at the very least, technologies used to connect may differ (e.g., cable and radio waves). Therefore, responsibilities are modularized and organized as a hierarchy, from lower to higher level. Each protocol in the stack takes care of an aspect of the communication and delegates the rest to the underlying layers.

  11. Socket   socket

    Internal endpoint for sending and receiving data at a single node in a computer network. Concretely, it is a representation of this endpoint in networking software (such as an entry in a table, listing communication protocol, destination, and so on), and is a form of system resource. Normally, a process requests a socket to the protocol stack (usually provided by the OS) and receives a socket descriptor, which is a handle that can be used to identify the socket (often, an integer). A socket is a local resource, which means that it cannot be directly referred from a different node in the network. It can be used to communicate with an arbitrary number of nodes, potentially even none (see datagram sockets), though usually it is exactly one.

    1. Internet socket

      Sockets are often used in conjunction with the Internet protocol suite. In this case they are assumed to be associated with a specific socket address, namely the IP address and port number for the local node (though it is not compulsory if the socket is only used to send data): this association is called binding and the socket address is the public address that can be used to send data to the socket from an external node. Internet sockets are also characterized by the transport protocol, so that two sockets bound to the same port with different protocols are in effect different sockets. A socket used to connect to another socket, such as in TCP, also has a remote socket address. This can be used to distinguish sockets, too. The local and remote sockets together are known as a socket pair.

      1. Types

        Depending on the protocol which is used, there are different socket types.

        1. Datagram socket   datagram udp

          Also known as connectionless socket, it uses the UDP protocol.

        2. Stream socket   stream tcp

          Also known as connection-oriented socket, it may use different protocols, including the TCP protocol.

        3. Raw socket   raw

          The transport layer is bypassed and the packet headers are made accessible to the application. Typically used in routers and other network equipment.

  12. Uniform Resource Identifier (URI)   uri

    It is a string of characters used to uniquely identify a resource. It can be thought as the name of the resource. Note that it tells nothing about where to find such resource.

  13. Uniform Resource Locator (URL)   url

    Reference to a web resource, specifying its location and how to access it. It is a specific type of identifier (URI). It can be thought as the address of the resource (with the additional constraint that addresses uniquely identify a resource).

  14. Uniform Resource Name (URN)   urn

    Historical name for a URI that uses the urn scheme, which basically is a rigid constraint to obtain uniqueness across space and time.

  15. Virtual Private Network (VPN)   vpn

    There are different types of VPNs, but the bottom idea is that communication between two machines in the network is tunnelled, usually to increase security or to add a mechanism of authentication and authorization in order to protect restricted resources.

    1. Tunnel   tunnel

      The main technology used by VPNs is a Virtual Network Adapter, which is used to add an additional layer to the normal protocol stack. This layer can be added at various levels, though normally it is either on top of layer 2 or 3 (in some cases, it may be on top of the application layer instead). Usually, the virtual network adapter encrypts the data and adds some specific headers, so on the receiving side there must be a corresponding virtual network adapter which is able to interpret and decrypt what it receives. This whole process is what is known as tunnelling. To encrypt data, normally some form of symmetric encryption is used: this usually requires an initial negotiation phase which is basically the process of connection to a VPN. This connection can also be used to set some other paramters, such as a virtual local IP address for the client.

    2. Types
      1. Site to Site

        Used to enable transparent communication between two different private networks over the Internet.

        1. Roadmap

          A detailed example of what happens:

          1. Say a user from site 1 on a device inside the network with IP 192.168.1.10 wants to send a message to a user in site 2 with a device with IP address 192.168.2.14.
          2. A packet will be generated on the sender's device as usual and as the packet is destined to the address 192.168.1.14 it will be forwarded to the network gateway 192.168.1.1.
          3. The gateway device will look at the destination and check its routing table where it will find that packets destined to 192.168.2.0/24 should be sent from vif0 interface.
          4. Listening on the vif0 interface will be a VPN application that will receive that packet.
          5. It will encrypt the packet, encapsulate with a custom VPN header (depending on the VPN protocol), and ship it out on eth0 with destination IP of site 2 (202.35.102.100) at port number 500.
          6. The packet will arrive at site 2's router. It will check that the packet is destined to itself at port 500. So it will forward the packet to the VPN server running in that device.
          7. Now the VPN server will strip off the outer VPN header, decrypt the packet and check the IP header of that packet.
          8. After finding out its destination address to be 192.168.2.14, it will send the packet out at eth1 where it will be finally be sent to the device 192.168.1.14.
      2. Remote User

        Used to enable access to a remote user to a private network.

        1. Roadmap

          A detailed example of what happens:

          1. The VPN client tries to connect with the VPN server and goes through all the authentication and authorization mechanisms.
          2. After passing all that, a session is created between the client and the server. The client gets a local IP address (say, 10.1.1.1) from the server to communicate with the network. The client chooses a random port number for communication e.g.3456.
          3. Now the client is good to send packets to any device in the internal network of its company. Say the client wants to send a message to a device in the remote network with IP address 192.168.1.14.
          4. A packet will be generated on the client’s device as usual and as the packet is destined to the remote network, it will be forwarded to vif0 by the client's routing table.
          5. Then the VPN client application listening on vif0 will encrypt the packet, encapsulate with a custom VPN header (depending on what VPN protocol we use), and ship it out on eth0 with destination IP of VPN server (202.54.34.58) at port number 500.
          6. The packet will arrive at company's router. It will check that the packet is destined to itself at port 500. So it will forward the packet to the VPN server running in that device.
          7. Now the VPN server will strip off the outer VPN header, decrypt the packet and check the IP header of that packet.
          8. After finding out its destination address to be 192.168.2.14, it will send the packet out at eth1 where it will be finally be sent to the device 192.168.1.14.
      3. Peer to Peer

        Used to create a private network over the Internet for a group of users.

        1. Roadmap

          A detailed example of what happens:

          1. Say user 1 wants to talk to user 3 using the virtual network.
          2. A packet will be generated in the user’s device and it will be destined to 192.168.1.12.
          3. According to user 1's routing table, such a packet needs to be forwarded through vif0 interface.
          4. vif0 will receive the packet, encrypt it and encapsulate it with a custom VPN header (depends on the protocol used), and ships it out through eth0 with the destination address of user 3's external IP address (162.155.57.99) and port number 500.
          5. User 3 will get the packet and see its destination port number and forward it to the VPN application that is listening on that port number.
          6. The VPN application will receive it, remove the outer header, decrypt it and send it to the vif0.
          7. From vif0 the OS kernel will find the destination address to be its own (192.168.1.12) and from there it will hand it over to the corresponding application expecting the packet.
      4. Secure Browsing

        Used to tunnel all the Internet traffic of a user through some server, to hide it from ISPs or other third parties.

        1. Roadmap

          This is basically the same as the remote user case. The difference is that all the traffic is routed to the LAN server. In a sense, the LAN server acts as a proxy.

      5. Virtual Private LAN service (VPLS)   vpls

        Used to extend an Ethernet network over a WAN.

        1. Roadmap

          This is similar to the site to site case, the difference being that the connection uses bridging and not routing. Every local network's gateway will know the list of MAC addresses for that area and will tunnel all requests with an unknown target. An example is:

          1. Say we have a network 192.168.1.0/24 extended over 3 site such that IP 1-100 are at site 1, 101-170 are at site 2 and 171-254 are at site 3.
          2. The VPLS application running at each branch will know about every MAC address in its branch.
          3. For unknown MAC address, it will be tunneled using any existing tunneling protocol over to other sites. Based on the replies from other connected sites the VPLS application will learn the MAC addresses and its origins (the same way as a normal bridge would learn).
          4. The unicast and multicast messages need not be forwarded over the WAN if source and destination devices are in the same site, but broadcast messages need to.
  16. Wide Area Network (WAN)   wan

    It is a telecommunications network or computer network that extends over a large geographical distance. Technically, the Internet is a WAN too.

1.11.3 Representational state transfer (REST)   rest

It is an architectural style used for distributed hypermedia systems, which happens to be the software architectural style of the World Wide Web. It can be described by a set of constraints (client-server, stateless, cache, uniform interface, layered system, code-on-demand) applied to elements (data elements, connectors and components) within the architecture.

  1. Elements
    1. Data Elements

      There are three fundamental options for transmitting data: process it and send a fixed-format image (hides information details from the client, but limits functionalities and puts a lot of load on the server), send both data and a rendering engine (hides information details from the client while retaining functionality, but can vastly increase the amount of information in a response) and send raw data along with metadata which describes it (puts the minimum possible load on the server, but loses the advantages of information hiding and requires that sender and recipient understand the same data types). REST focuses on the third option, but limiting the scope of what is revealed to a standardized interface. A representation of a resource is transmitted in a format matching one of an evolving set of standard data types.

      1. Resource

        Any information that can be named can be a resource. A resource R is a temporally varying membership function M(t), which for time t maps to a set of entities, or values, which are equivalent. Therefore, a resource is a conceptual mapping to a set of entities, not the entity that corresponds to the mapping at any particular point in time. The values in the set may be resource representations and/or resource identifiers.

      2. Resource Identifier

        The unique identifier of a resource.

      3. Representation

        A representation (of a resource) is a sequence of bytes, plus metadata to describe those bytes.

      4. Representation Metadata

        Information in the form of name-value pairs, where the name corresponds to a standard that defines the value's structure and semantics.

      5. Resource Metadata

        Metadata about the resource which is not specific to a particular representation.

      6. Control Data

        Defines the purpose of a message between components.

    2. Connectors

      They encapsulate the activities of accessing resources and transferring resource representations and they present an abstract interface for component communication.

      1. Client

        The agent that initiate a communication by sending a request.

      2. Server

        An agent that listens for connections and responds to requests in order to supply access to its services.

      3. Cache

        Deals with the caching of responses. Can be both client-side and server-side. It can also be shared.

      4. Resolver

        Translates partial or complete resource identifiers into the network address information needed to establish an inter-component connection (as is the case with DNS).

      5. Tunnel

        It simply relays communication across a connection boundary.

    3. Components

      The main actors of the communication.

      1. User Agent

        A user agent uses a client connector to initiate a request and becomes the ultimate recipient of the response. The most common example is a Web browser.

      2. Origin Server

        An origin server uses a server connector to govern the namespace for a requested resource. It is the definitive source for representations of its resources and must be the ultimate recipient of any request that intends to modify the value of its resources. Each origin server provides a generic interface to its services as a resource hierarchy. The resource implementation details are hidden behind the interface.

      3. Proxy   proxy

        A proxy, or forward proxy, component is an intermediary selected by a client to provide interface encapsulation of other services.

      4. Gateway   gateway reverse proxy

        A gateway (or reverse proxy) component is an intermediary imposed by the network or origin server to provide an interface encapsulation of other services. Basically, it is a server that retrieves resources on behalf of a client from one or more servers.

  2. Constraints
    1. Client-Server

      This is motivated by the need for a separation of concerns, which allows components to evolve independently and promotes portability of user interfaces and scalability of server components.

    2. Stateless

      Session state is kept entirely on the client. This improves visibility (all data is contained in each request), reliability (possibility to recover from partial faults) and scalability (reduced load on servers). The drawbacks are that more data must be transmitted for each request and that servers should rely on correct client implementations.

    3. Cache

      Data within a response must be implicitly or excplicitly labeled as cacheable or non-cacheable. This improves efficiency, scalability, and user-perceived performance. The drawback is that it may decrease reliability in case of stale data.

    4. Uniform Interface

      All components communicate according to a fixed interface. This is the central feature of REST, it decouples implementations from the services they provide, encouraging independent evolvability. The drawback is that it degrades efficiency, since information is tranferred in a standardized way instead of an application-specific way.

    5. Layered System

      Functionalities can be divided in hierarchical layers, and each component cannot "see" beyond the immediate layer with which it is interacting. The main drawback is that this adds overhead and latency to the processing of data.

    6. Code-On-Demand

      Client functionality can be extended by downloading and executing code in the form of scripts. This improves extensibility but reduces visibility, so it should be seen as an optional constraint (it may be taken into account in the design of a subsystem, but with the understanding that in other subsystems may be unavailable).

1.11.4 Security   security

  1. Clickjacking

    Malicious technique of tricking a user into clicking on something different from what the user perceives they are clicking on. Usually this is done using invisible frames on top of the real page. This is particularly effective when frames contain respectable pages that the user may be logged in (such as Facebook or Amazon), as in this way the attacker may indirectly gain the same user privileges.

  2. Cross-site request forgery (CSRF or XSRF)

    Technique that enables a malicious website to send unauthorized commands to a different website. This can happen when an attacker finds a link that reproduces a certain action if the user is logged to certain site (for instance a link to a banking website that executes a fund transfer). Since the browser normally sends the session cookie with every request, the attacker could pretend to be the user and executes illegitimate actions. A browser-based mitigation is the custom of blocking cross-domain requests (that is, a client script cannot request content from a different domain than the one that produced the script itself).

    1. CSRF token

      A way to deal with this attack while still allowing cross-domain requests is the generation of a token which is added to every page served to the user. The token is user-specific or short-lived or both and the attacker has no way of guessing it.

    2. Cross-origin resource sharing (CORS)

      Mechanism that solves the same problem that the CSRF token solves, but in a different way. The standard describes new HTTP request headers that the browser should set. Briefly, before attempting a dangerous request the browser should send an OPTIONS request with the Origin HTTP header set, so that the server can answer whether that origin is allowed to perform such requests. This is the solution endorsed by W3C.

    3. iframe

      Iframes are considered separate components, so they are allowed to make cross-origin requests. The CSRF attack prevention, in this case, is implemented forbidding communication between iframes and the parent document. Again, there are ways to circumvent this limitation, but it is not straightforward.

  3. Cross-site scripting (XSS)

    Malicious technique that enables attackers to inject client-side scripts into web pages viewed by other user. This can happen when a server delivers content (private or public) without filtering user input: an attacker can insert script tags in the code using this technique and then run code with the same privileges of the original code (using this, for instance, to steal authentication cookies).

  4. Session fixation

    Malicious technique that enables one person to fixate another's person identifier. It typically happens when a session identifier is allowed to be sent as a GET parameter or as POST data. A simple scenario is an attacker providing a link which contains a random session identifier to a vulnerable site: the victim will click the link and log in as usual, and the attacker will know the session identifier of an active session. A possible solution is to regenerate the session identifier upon successful login.

  5. Same-origin policy

    A web browser permits scripts contained in a web page to access data in a second web page only if both pages have the same origin (defined as a combination of URI scheme, hostname and port number). This restriction is necessary because otherwise a script running in a malicious web page could request information from any page where the user is logged in (like, for instance, a banking site). This is known as cross-site request forgery.

  6. Single sign-on (SSO)   sso

    Property of access control of multiple related, yet independent, software systems. With this property, a user logs in with a single ID and password to gain access to multiple systems.

1.11.5 Service-oriented architecture (SOA)   soa

Style of software design where services are provided by application components through a communication protocol over a network. It emphasizes the distributed and separate maintenance of the deployed services. There are no industry standard implementations, but one common approach is to use web services (for instance, SOAP). Possible issues are increased overhead, coupling not loose enough, managing of metadata, no uniform testing framework (due to heterogeneity and complexity).

  1. Enterprise service bus (ESB)   esb

    Very generic expression that should more or less designate an implementation of the abstract concept of service broker. It is called bus as an analogy to the bus found in computer hardware architecture (component used to transfer data between other components).

  2. Main components
    1. Service provider

      It creates services and provides their information to the service registry.

    2. Service broker

      Also called service registry or service repository, its main functionality is to make services available. The scope of a broker can be public or private.

    3. Service requester/consumer

      It locates entries in the broker registry using various approaches, then binds to the service provider in order to invoke one of the services.

  3. Microservices   microservice

    They are closely related to SOA, but there is emphasis over automated deployment and management (and to DevOps in general). Also, often they use REST as communication protocol, while traditional SOA requires more precise contracts (and in fact often uses SOAP).

  4. Service

    In this context, a service is a discrete unit of functionality which:

    1. logically represents a business activity with a specified outcome;
    2. is self-contained;
    3. is a black box for its consumers;
    4. may consist of other underlying services.

1.11.6 Simple Object Access Protocol (SOAP)   soap

Protocol specification for exchanging structured information in the implementation of web services in computer networks, using XML as format.

  1. Relation with WSDL

    The WSDL describes a web service, while SOAP describes how to communicate with the service and what messages can be exchanged. In particular, the WSDL will specify whether a service understands the SOAP protocol.

  2. Terminology
    1. Protocol concepts
      Term Description
      Node Physical or logical machines which are used to transmit, forward, receive or process SOAP messages.
      Role All nodes have a specific role, which limits what they can do.
      Protocol Binding The communication underlying protocol (for instance, TCP).
    2. Data encapsulation concepts
      Term Description
      Message The information being exchanged.
      Envelope The enclosing element of a message, which identifies it as a SOAP message.
      Header block A set of instructions targeted to a node with a specific role. A message generally contains more than one block.
      Header A collection of one or more header blocks.
      Body The body of the message. The interpretation and processing of it is defined in the header.
      Fault Optional information about an error. If present, it is a child of the body element.
    3. Message sender and receiver concepts
      Term Description
      Sender The node that transmits the message.
      Receiver The node that receives the message.
      Message path The set of all the nodes traversed to deliver the message.

1.11.7 Programming

  1. Asynchronous JavaScript and XML (AJAX)   ajax

    AJAX is a technique through which it is possible to update parts of a web page, without reloading the whole page. Tipically, background processes implemented through JavaScript fetch data from the server using SOAP requests and then update the page (again, using JavaScript) in real time. This can greatly enhance the response time and reduce the load on the server as it is not necessary to fully reload the page for every minor change. One of the shortcomings is that careless use of AJAX can interfere with the built-in history management of browsers (a change in the page does not correspond necessarily to a press of the back button). Possible solutions are careful design or manual tuning of the browser history.

  2. Cascading Style Sheets (CSS)   css

    It is a style sheet language used to describe the presentation of documents written in markup languages (usually html, but could be XML or other). The "cascading" refers to the language feature that solves conflicting attribute assignments according to the rule of "later changes prevail" (in reality, weights are assigned to every assignment according to some specific rules).

    • margin: shorthand for margin-top margin-right margin-bottom margin-left. If used with the auto option, it horizontally centers the element within the container.
    • padding: same as margin, but the space is taken from inside the border of the element. Does not have an auto option.
    • float: none|left|right|initial|inherit lets an element float. Ignored by absolutely positioned elements.
    • position: static|absolute|fixed|relative|initial|inherit. Static is default, absolute is positioned relative to its first not static ancestor element. Fixed is positioned relative to the viewport. Relative is positioned relative to its normal position.
    1. Common errors
      • Don't forget to escape the colons!
  3. Document Object Model (DOM)   dom

    It is a cross-platform and language-independent standard used to access documents, in particular HTML and XML. It treats documents as tree structures.

  4. HyperText Markup Language (HTML)   html

    Core attributes: id, title, class, style. The id attribute is used to identify an element. The title attribute is used for hovering suggestions. The class attribute is used to associate an element with a style sheet (CSS). The style attribute lets devolopers add style content directly inside the html page (should be used as little as possible). Internationalization attributes: dir, lang, xml:lang. The dir attribute set the direction the text should flow to (ltr for left to right and rtl for opposite). The lang (kept for backward compatibility) and xml:lang attributes are both used to set the main language of the page.

  5. Java   java
    1. Facelets

      Open source Web template system and default view handler for JSF 2.0. It supports all JSF UI components.

      1. Templating

        Mainly there are four tags which are used for this purpose.

          This is used in the template file. It defines content that is going to be replaced. It supports the name attribute to specify a name. Usually default content is placed inside this tag.

        2. ui:define

          Defines content that is going to replace a ui:insert element. It supports a name attribute which must match the name attribute of the ui:insert element.

          Includes content from another file, specified through the src attribute. If ui:composition is used in the src file, all content outside this tag is ignored (useful to not include headers or other descriptive content which is not needed).

        4. ui:composition

          Defines a group of elements that can be included somewhere using ui:include. Otherwise, if used with the template attribute, it loads the specified template; then child ui:define elements can be used to replace parts of the template with custom content.

      It is a standard. It describes a class that must have only private variables and a public no-argument constructor, and that must be Serializable. Beans are widely used in frameworks because of this guarantees. JSF uses them to implement the model.

    4. JavaServer Pages (JSP)

      It is a Java view technology running on server machine to write template text. Using taglibs supported by Java code the output can be changed dynamically. The servlet container compiles the JSP at startup into a class extending HttpServlet. Files have extension jsp.

        This attribute is used on the server side to tell which UIComponents must be processed during the entire JSF lifecycle in a form (only input fields and command fields can be included, and they must have an id).     6. Servlet   servlet

      Java API running on server machine. Intercepts requests made by client and sends responses. One of the most popular is the HttpServlet, used to intercept HTTP methods such as GET and POST. Servleton stackoverflow.     1. Django   django

      High-level Python Web framework. It is structured along a classic MVC pattern, though the view part is still an abstraction more similar to a controller. The basic flow is that an HTTP request comes for a particular URL, Django uses some rules to decide which view should deal with it, then the view processes the request and returns an HTTP response.

      1. App   app

        Django encourages to structure code into apps, where one project can have many. These are the fundamental reusable units of a Django project. python manage.py startapp app_name is the command to use to create a new app. It uses the current folder.

    2. Web Server Gateway Interface (WSGI)   wsgi

      Standard specification for a universal interface between web servers and web applications or frameworks (server-side). It has two sides, the server side and the application side. To process a request, the server executes the application and provides environment information and a callback function to it. The application then processes the request, returning the response to the server using the callback function. Finally, the server will forward it to the client. The interface can also be implemented by a middleware that implements both sides of the API and mediate the communication between server and application.

1.11.8 Web portal   portal

A specially designed web site that brings information together from diverse sources in a uniform way. It often features additional services (such as e-mail, news, chats…) and an internal search engine. Often, portals are made by enterprises, both for internal and external use. When internal, the proper name is enterprise portal.

  1. Self service portal

    A portal which enables users to perform transactions without the aid of a customer representative.

  2. Portlet

    A pluggable user interface software component, managed and displayed in a portal. Typically a portal page is a collection of non-overlapping portlet windows, each displaying a different portlet. This is the way portals manage dynamic content. There are standards for portlets, that ensure compatibility with portals supporting those standards.

1.11.9 Web Server   server

Computer system that processes requests via http or, more specifically, the software that accepts and supervises the http requests. The primary purpose of a server is to store, process and deliver content to clients. The content may simply be a file on the server memory, which the client asks for using its relative path (relative to the root directory of the server, which is never the root directory of the system). It could also be a virtual path, that the server software is able to understand and process. The final result, in both cases, will be an http response.

  1. Application server   server

    Often, the terms web server and application server are used as synonyms, but there are key differences.

    Web server Application server
    Designed to serve HTTP content. Not limited to HTTP.
    Mostly static content, though there are plugins for Perl, PHP, … Dynamic content.

    Note that an application server often is built upon a web server and extends its functionality. Sometimes web servers are used as reverse proxies to app server: they serve static content and transparently forward dynamic requests to app servers.

  2. Kernel-mode vs User-mode

    A server can be either incorporated in the OS kernel or in user space (like other regular applications). To increase efficiency, the former solution is preferred.

  3. Load limits

    A server can only handle a limited number of concurrent client connections per IP address and TCP port. In addition, it can only handle a maximum number of requests per second, depending on many factors (including hardware).

  4. Server Cluster   cluster

    A group of two or more servers that work together. Usually, load balancers sit in front of server clusters and decide how to distribute requests among the different servers. Past a certain point no single server can be as powerful as a cluster. In addition, having a cluster is useful for redundancy, ease of maintenance and so on.

1.12 Programming languages

1.12.1 C++   c

  1. Basic definitions   theory
    1. Class and struct   class struct

      The members and base classes of a struct are public by default, while in class, they default to private. Note: you should make your base classes explicitly public, private, or protected, rather than relying on the defaults.

      Struct and class are otherwise functionally equivalent.

    2. Attribute   attribute

      Additional metadata that can be though of as comments for compilers. For example, an attribute can specify that a function will never return via normal execution flow (because it always throws an exception).

      C++ attributes are similar to Java annotations, but they are compiler-defined (so users cannot define them, and they cannot have effect at runtime). They appeared as vendor-specific compiler extensions and were later standardized (since C++11), though the non-standard version is still popular.

      1. Visibility   visibility

        IBM and GNU attribute that can be used to control whether a structure/union/class or enumeration that is defined in one module can be referenced or used in other modules. It only affects types with external linkage and cannot be used to increase the visibility.

        The syntax is __attribute__((visibility("default"))), where default can also be protected, hidden and internal.

    3. Declarations and definitions   declaration definition

      Declarations introduce (or re-introduce) names into the C++ program. Definitions are declarations that are sufficient to use the entity identified by the name. A declaration is one of the following:

      • Function definiton,
      • Template declaration,
      • Explicit template instantiation,
      • Explicit template specialization,
      • Namespace definition,
      • Linkage specification,
      • Attribute declaration,
      • Empty declaration,
      • A function declaration without a decl-specifier-seq,
      • Block declaration.

      The complete reference is here.

      1. Declaration specifier   specifier declspecifierseq typespecifierseq

        Declaration specifiers (decl-specifier-seq) is a sequence of the following whitespace-separated specifiers, in any order:

        • typedef specifier.
        • Function specifier (inline, virtual, explicit).
        • friend specifier.
        • constexpr specifier.
        • Storage class specifier (register, static, thread_local, extern, mutable).
        • Type specifiers (type-specifier-seq):
          • class specifier.
          • enum specifier.
          • Simple type specifier (primitive, auto, class name, enum name, …).
      2. Simple declaration

        A simple declaration is a statement that introduces, creates, and optionally initializes one or several identifiers, typically variables.

    4. Inheritance   inheritance

      Use colons after a class name. Access specifiers can be used to reduce the visibility of all public and protected members of the base class.

    5. Library   library

      Basically, libraries can be static (embedded in the binary executable) or dynamic (loaded in memory at runtime). The dynamic linking is taken care of by the operating system. Therefore, there are different approaches.

      1. Shared Object   sharedobject

        A library that is automatically linked into a program when the program starts, and exists as a standalone file. The library is included in the linking list, so it must be present at compile time and when the application starts.

        This is a Linux-specific object. The advantage over a static library is that the version of the library can be modified without recompiling the program, and that the executable will have smaller size. On the other hand, it introduces dependency requirements.

      2. Static library   staticlibrary

        A library that is merged into the actual program itself at build time. It must be present at compile time, but it does not have to be present at runtime, cause it is embedded in the binary executable.

      3. Dynamic Link Library (DLL)   dynamiclinklibrary dll

        Similar to a shared library, but for Windows. At compile time, usually the linker needs a LIB object with the definitions to properly link. At runtime, the DLL is needed instead.

        Windows requires two different objects because, historically, it was common to compile a program in a different environment from the one where the program would be run. In Linux instead, the philosophy was "compile where you run".

    6. Linkage   linkage

      If a name has linkage, it refers to the same entity as the same name introduced by a declaration in another scope.

      • No linkage: the name can be referred to only from the scope it is in.
      • Internal linkage: the name can be referred to from all scopes in the current translation unit.
      • External linkage: The name can be referred to from the scopes in the other translation units.
    7. Name   name
      1. Dependent   dependent

        A name that depends on a template parameter, as in vector<T> vt; (vector<T> is the dependent name and its type is a dependent type). All other names are non-dependent. typedef transfers the quality of being dependent: typedef T another_name_for_T; makes another_name_for_T a dependent name.

      2. Qualified   qualified

        A name that specifies a scope, as in std::cout.

      3. Unqualified   unqualified

        A name that does specify a scope, as in cout (assuming using std::cout has been used).

    8. Namespace   namespace

      Namespaces provide a method for preventing name conflicts in large projects. To create a namespace, normally the syntax is namespace NAMESPACE_NAME {...}. To reference definitions in a namespace, use NAMESPACE_NAME::DEFINITION.

    9. Object   object

      C++ programs create, destroy, refer to, access and manipulate objects. An object is a region of storage that has

      • size
      • alignment requirement
      • storage duration
      • lifetime (bounded by storage duration or temporary)
      • type
      • value (may be indeterminate)
      • optionally, a name
    10. Pointer variable and reference variable   pointer reference

      The main differences are the following (there are other, less important ones):

      • A pointer can be re-assigned and a reference cannot (and it must be assigned at initialization). 

        int x = 5;
int y = 6;
int *p;
p =  &x;
p = &y;
*p = 10;
assert(x == 5);
assert(y == 10);

int x = 5;
int y = 6;
int &r = x;

      • A pointer has its own memory address, while a reference has the same address as the original variable. This implies that the real address of a reference will not be revealed. 

        int x = 0;
int &r = x;
int *p = &x;
int *p2 = &r;
assert(p == p2);

      • There can be pointers to pointers, whereas references only offer one level of indirection. 

        int x = 0;
int y = 0;
int *p = &x;
int *q = &y;
int **pp = &p;
pp = &q;//*pp = q
**pp = 4;
assert(y == 4);
assert(x == 0);

      • Pointers can be assigned nullptr directly, whereas references cannot (but it is possible to manage trying hard enough). 

        int *p = nullptr;
int &r = nullptr; <--- compiling error
      • Pointers need to be dereferenced to access the memory location, whereas references can be used directly (thus making them pretty much aliases of the original variables).
    11. Scope   scope

      Each name that appears in a program is only valid in some possibly discontiguous portion of the source code called its scope. Reference.

      1. Block   block

        The scope associated to compound statements, functions, exception handlers, and to constructs such as for loop, if statement, while loop, or switch statement.

      2. Class   class

        The class scope includes the rest of the class body and all function bodies (even if defined outside or before).

      3. Function prototype   function

        The scope of a name introduced in the parameter list of a function declaration which is not a definition. It ends at the end of the function declarator.

      4. Namespace   namespace

        The scope of a name defined in a namespace encompasses all definitions of the namespace, starting from the point of declaration of the name. It also includes all namespaces where a using directive is used. The top-level scope of a translation unit is also a namespace, properly called global namespace scope (but also file scope or global scope).

    12. Special member functions   function

      Member functions which are defined by the compiler even if not defined in a class. They are:

      • Default constructor,
      • copy constructor,
      • move constructor (since C++11),
      • copy assignment operator,
      • move assignment operator (since C++11),
      • destructor.
      1. Constructor   constructor

        Special member function of a class that is executed whenever a new object of that class is created. Note that the default constructor with no arguments is called when writing MyClass c;.

        1. Initialization list   initialization colon

          It can be used to call base class constructors or to initialize member variables before the body of the constructor executes (useful to initialize const members, which would otherwise not be possible). The syntax is ClassName(...) : BaseClassName(...) {...}, or ClassName(...) : v1(val1), v2(val2), v3(val3) {...}. It is also a convention to use it in general to initialize variables.

        2. Syntax

          ClassName(...) : initialization-list.

      2. Copy constructor   copyconstructor

        Special constructor that is used to make a copy of an existing instance. The signature is MyClass( MyClass& other) (possibly with const and/or volatile qualifiers). Copy constructors are automatically created by the compiler if not defined, and by default they perform a member-wise copy. It is good practice to use const in a copy constructor, to avoid tricky compilation errors.

        1. Example

          MyClass c = MyClass(); performs copy initialization.

      3. Move constructor   moveconstructor andand

        A non-template constructor of class T whose (basically) only parameter is T&& (with or without const and/or volatile). It is used with rvalue references.

        Its purpose is to "steal" resources from a temporary object (which has to be left in a valid state!).

      4. Copy assignment operator   assignment copyoperator

        Operator which assigns an object to another of the same type. The signature is of the form MyClass& operator=(const MyClass& rhs), but there are many variants. Assignment operators are automatically created by the compiler if not defined, and by default they perform a member-wise copy.

        1. Example

          MyClass c1; MyClass c2; c1 = c2; use the assignment operator.

      5. Move assignment operator   assignment moveoperator

        A non-template non-static member function with the name operator= that takes exactly one parameter of type T&& (with or without const and/or volatile).

        Its purpose is basically the same of a move constructor.

      6. Destructor   destructor

        Special member function of a class that is called when the lifetime of an object ends. Its purpose is to free resources.

        1. Syntax

          ~ClassName().

        2. Rule of Three   three

          If you define the destructor, copy constructor or copy assignment operations then define all three of them. This is because the compiler defaults will no longer be valid for the rest. Moreover, a virtual destructor should only be defined on an abstract class (also, non-abstract classes should not be used as base classes).

        3. Rule of Zero   zero

          Do not handle resource management. Use classes such as std::vector, std::string, std::unique_ptr and std::shared_ptr instead. In this case, there is no need for destructors.

    13. Storage duration   storageduration

      All objects in a program have one of the following durations:

      • automatic: the object is allocated at the beginning of the enclosing code block and deallocated at the end. This is the default.
      • static: allocated at the beginning of the program and deallocated at the end. Only one instance exists. All objects declared at namespace scope have this duration, plus those declared with static or extern.
      • thread: allocated when the thread begins and deallocated when it ends. Each thread has its own instance. Only objects declared thread_local have this duration.
      • dynamic: allocated and deallocated per request using dynamic memory allocation.
    14. Translation unit   translationunit

      Basic unit of compilation in C++. It consists of the contents of a single source file, plus the contents of any header files directly or indirectly included by it, minus those lines that were ignored using conditional preprocessing statements.

    15. Value category   valuecategory lvalue rvalue

      One of the two (independent) properties that characterize a C++ expression, the other being its (non-reference) type. Every expression can be a prvalue, xvalue or lvalue (exclusive OR).

      Actually, the most important value categories are lvalue and rvalue. The simple definition is that an lvalue represents an object that occupies some identifiable location in memory (i.e. has an address), while rvalue represents all the rest (no address, just stored in some temporary register). Generally speaking, lvalues can be modified and rvalues not, but since C++11 this is not entirely true.

      1. lvalue   lvalue

        lvalue originally stood for suitable for left-hand-side of assignment. const variables though are lvalue expressions that cannot be assigned, so it now stands for locator value. An lvalue that can be assigned is called modifiable.

      2. rvalue reference   rvalue andand moveconstructor

        Since C++11, move semantics was added to the language. The idea is that when an rvalue expression is being passed to a function, there is no need to deep copy it, because it will be discarded anyway. So a move constructor can be called instead, saving the time for a copy. This call is done automatically when appropriate.

  4. Enumeration   enum

    An enumeration is a user-defined type that consists of a set of named integral constants known as enumerators. It can be scoped or unscoped (depending on whether the class keyword is used): scoped enumeration must be qualified to be used. To cast to int, use static_cast<int>. 

    // unscoped enum
enum [identifier] [: type]
  {enum-list};

// scoped enum
enum [class|struct]
[identifier] [: type]
  {enum-list};

// type can be any integral type

    enumerators are automatically assigned integral values, if not explicitly done by the programmer. Their default value is one plus the value of the preceding enumerator, starting from zero. Note that integral values may be repeated.

  5. Interface   interface

    C++ does not really have interfaces, it just uses abstract classes (since it supports multiple inheritance, interfaces are not really needed). An abstract class is a class containing at least one pure virtual function. Note that, of course, abstract classes cannot be instantiated.

  6. Pointer   pointer
    1. Smart pointer   smart std

      A smart pointer is an object that wraps a raw C++ pointer and manages the lifetime of the object pointed to. The use of smart pointers instead of raw pointers is encouraged by the rule of zero.

      1. Choosing the smart pointer to use
        • 1.12.1.6.1.3 should be used when there are not multiple references to the pointed object. This should be the default choice.
        • 1.12.1.6.1.2 should be used when an object must be referenced from multiple places and it must stay in scope as long as at least one of these places needs it.
        • 1.12.1.6.1.4 should be used for references that do not need to take ownership of an object. In particular, it should be used to break circular references.
      2. std::sharedptr   shared

        A smart pointer that shares the ownership of a pointed object with other shared pointers: the object is destroyed when the last shared pointer is destroyed or reset.

        1. Circular reference

          Circular references with shared pointers cause memory leaks: 

          struct Owner {
   boost::shared_ptr<Owner> other;
};

std::shared_ptr<Owner> p1 (new Owner());
std::shared_ptr<Owner> p2 (new Owner());
p1->other = p2; // p1 references p2
p2->other = p1; // p2 references p1

// Oops, the reference count of of p1 and p2 never goes to zero!
// The objects are never destroyed!
      3. std::uniqueptr   unique

        A smart pointer that is the sole owner of the pointed object. Unique pointers cannot be copied, but they can transfer ownership to another pointer using std::move.

        1. Return from function

          Unique pointers can be returned from functions thanks to the process of copy elision. See this answer.

      4. std::weakptr   weak

        A smart pointer that does not own the pointed object. The pointed object can be destroyed even if there are weak pointers pointing to it. A weak pointer can acquire temporary ownership by being converted to a 1.12.1.6.1.2. Weak pointers are used to break circular references.

  7. Polymorphism   polymorphism

    For reasons related to low-level optimization, in C++ an object always has a fixed type and size known at compile time (plus a fixed address, if applicable). If a virtual function is called on an object, the version associated to its compile time type is called (usually, virtual functions are implemented as a per-object pointer to a table of functions which is unique to each type: that table is used to resolve the call).

    To use polymorphism as it is seen in other languages (see Java), the way to go is to use references and pointers. In this case, C++ guarantees that the version of a virtual function to use will be decided depending on the run time type of the object the reference/pointer points to.

  8. Preprocessor   preprocessor

    Component of the compiler which transforms source files before the actual compilation starts. It is often a separate program invoked by the compiler as the first part of translation. Preprocessor directives starts with # and do not end with a semicolon.

    1. Macro   macro define

      A macro is a fragment of text which has been given a name: whenever the name is used, the preprocessor will replace it with the corresponding text. The replacement process is symbolic but some basic logic can be added to it.

      Macro definitions always start with #define and, by convention, the name should be in capital letters. If spanning across multiple lines, \ must be put at the end of each line except the last one. After the replacement, everything will be on the same line.

      There are a few predefined macros, useful in multiple contexts, such as =__FILE__= and __LINE__, which expand to the name of the current input file and the current line number, respectively.

      1. Object-like macro

        Syntax is #define macro-name replacement-text, with replacement-text possibly omitted (in which case the preprocessor will simply erase macro-name).

      2. Function-like macro   function

        Macros that behave similarly to function calls. The syntax is #define macro-name([args]) replacement-text, with replacement-text possibly omitted. Note that there cannot be a space between macro-name and the parenthesis. To use the macro, use it as a function, passing the correct number of arguments (empty arguments are allowed). Do not put a semicolon at the end, though.

        Function-like macros can also have a variable number of arguments (using __VA_ARGS__).

        1. Pitfalls

          Using function-like macros can be tricky, particularly when passing complicated arguments.

          • Always surround an argument with parenthesis, if applicable, to prevent operators precedence issues.
          • C++ syntax is not relevant to the preprocessor: passing array[x,y] is like passing array[x and y.
        2. Stringizing

          If # is added in front of a macro argument, the argument is converted to a string. No further macro expansion happens, the compiler will later find a string which corresponds exactly to the macro argument. Special characters will be escaped if contained in a string constant (for example, \n = "foo\n" translates to \n = \"foo\\n\").

        3. Concatenation

          ## is used to concatenate two arguments. Again, the concatenation happens at a symbolic level.

      3. #undef   undef

        Used to undefine a macro from its name (#undef MACRO-NAME).

    2. #include   include

      The preprocessor replaces the text with the content of the specified file. #include "filename" is used when including source files (the preprocessor scans the current directory), while #include <filename> is used to include standard library header files (the preprocessor scans implementation-dependent directories).

      1. #include guard   include guard

        Particular construct used to avoid the problem of double inclusion. The standard approach is 

        #ifndef NAME_H
#define NAME_H

struct foo {
    int member;
};

#endif /* NAME_H */

        but it can be complicated to ensure that the chosen names do not collide. So many implementations provide a #pragma once directive that, inserted at the top, ensures that a file is included only once.

    3. Conditional compilation   if conditional

      #if, #ifdef, #ifndef, #else, #elif and #endif can be used for conditional compilation. The preprocessor skips that part entirely.

  9. Slicing   slicing

    Copying a derived object on a base object results in slicing, which means that only information about the base class is retained. This could be done differently, but it would pose problems when the objects have different sizes, as C++ reserves memory for an object based on its compile time type, not its run time type.

    1. Non-trivial risk of slicing
      // let B derive from A
B b1;
B b2;
A& a_ref = b2;
a_ref = b1;
// b2 now contains a mixture of b1 and b2

      This behavior is caused by the fact that the assignment operator is not virtual, so a_ref = b1 uses A's assignment operator.

  10. std   std
    1. initializerlist   initializerlist

      An object of type std::initializer_list<T> is a lightweight proxy object that provides access to an array of objects of type const T. It can be constructed using the syntax {e1, e2, e3...}. It is useful to quickly construct arrays and passing them to objects.

  11. Symbols and keywords
    1. Keywords   keyword
      1. auto   auto

        For variables, specifies that the type of the variable that is being declared will be automatically deduced from its initializer. For functions, specifies that the return type is a trailing return type or will be deduced from its return statements. It was introduced in C++11. The automatic deduction works more or less as the automatic deduction of parameterized types.

      2. const   const

        Type qualifier. It normally applies to whatever is to the left.

        1. Variable   variable

          const int CONSTANT = 46 is used to declare a named constant.

        2. Pointer   pointer

          const int * CONSTANT or int const * CONSTANT define a pointer to a constant integer. int * const CONSTANT defines a constant pointer to a variable integer. int const * const CONSTANT defines a constant pointer to a constant integer.

        3. End of function   function

          At the end of a function, it means that the function cannot change the value of any class variable, unless that variable is mutable.

      3. constexpr   constexpr

        A specifier that declares that a variable or a function can be used in a constant expression, which is basically an expression that can (but does not have to) be evaluated at compile time.

        1. Variable   variable

          Roughly speaking, it must be immediately initialized. In this case, constexpr implies const.

        2. Function   function

          The definition of functions that can be marked as constexpr is complicated. In any case, in this case it implies inline.

      4. default   default

        Since C++11, default can be used to explicitly tell to the compiler to use the default version (that is, the one that is generated) of a special member function. Otherwise, it is used in switch constructs.

      5. explicit   explicit

        Used to tell the compiler that a single-argument constructor cannot be used for implicit conversion. Otherwise, a void doBar (Foo foo) function could be called with doBar(1); if Foo has a Foo (int foo) {...} constructor: the compiler would automatically and silently take care of the conversion.

        Best practice is to always use explicit when possible.

      6. inline   inline

        A specifier that declares a variable or a function to be able to break the One Definition Rule (ODR) and be defined in multiple translation units. This is usually required for definitions contained in header files. Note that if the definition is contained in a class definition, the keyword is implicit.

        It is up to the programmer to ensure that all the definitions work well together.

        1. Historical use

          Historically, inline was a strong compiler hint to inline a function, that is, to replace a function call by repeating the function body directly at the call site. As compilers became better at optimizing, they started ignoring the hint.

          Notice that since compilers process one file at a time, they need the definition inside the file to be able to inline a function. So using the inline keyword may still be a good practice for functions that should be inlined. Also, notice that smart linkers performing Link-Time Code Generation (LTCG) may be able to inline functions that the compiler could not, but this kind of optimization is expensive on large projects so it may be better not to rely on it.

      7. new   new

        Used to dynamically allocate memory. One of the main reasons is to have arrays of variable size, but it is not necessary if std::vector is used.

      8. typedef   typedef specifier

        It creates an alias that can be used anywhere in place of a type name. Hint: to understand what alias is being defined, just read the declaration as if typedef is not there. The scope of the alias follows the normal rules for variables. 

        // simple typedef
typedef unsigned long ulong;

// the following two objects have the same type
unsigned long l1;
ulong l2;


// more complicated typedef
typedef int int_t, *intp_t;
// which is equivalent to
typedef int int_t;
typedef int *intp_t;


// common C idiom to avoid having to write "struct S"
typedef struct {int a; int b;} S, *pS;

// the following two objects have the same type
pS ps1;
S* ps2;

// typedef can be used anywhere in the decl-specifier-seq
long unsigned typedef int long ullong;
        1. struct, union, enum   struct union enum

          The C language standard mandates separate namespaces for different categories of identifiers, including tag identifiers (for struct, union, enum) and ordinary identifiers (for typedef and other identifiers). That is why in C there is the trick of typedef'ing structures in order to omit the struct keyword when using them. In C++ this is not so relevant, cause all struct, union, enum, class declarations act like they are implicitly typedef'ed, as long as the name is not hidden by another declaration with the same name. For more details see here.

      9. typename   typename

        It basically tells the compiler that what follows is a type. It should be used in front of qualified, dependent names. typename can also be used in templates, but that use is completely unrelated and just introduced because some people did not like to use class.

        1. Motivation

          T::iterator * iter; could be a pointer to a nested type iterator, but it could also be a product between iter (which may exist or not) and a static int iterator. There is no way to distinguish, so typename must be used in the first case.

      10. using   using namespace

        A directive that tells the compiler what namespace is being used. using namespace NAMESPACE_NAME imports all definitions contained in NAMESPACE_NAME. using NAMESPACE_NAME::DEFINITION imports DEFINITION.

        1. typedef   typedef

    4. Substitution Failure Is Not An Error (SFINAE)   sfinae

      When invalid substitution of template parameters happens, it does not necessarily produce an error. In particular, if a suitable overload version is found, broken overloads are simply discarded by the compiler. The original motivation was to avoid creating ill-formed programs when unrelated template declarations were visible, but it has also become a way to implement compile time introspection.

      1. Basic example
        struct Test {
    typedef int foo;
};

template <typename T> 
void f(typename T::foo) {} // Definition #1

template <typename T> 
void f(T) {}               // Definition #2

int main() {
    f<Test>(10); // Call #1.
    f<int>(10);  // Call #2. Without error (even though there is no int::foo) thanks to SFINAE.
}
      2. enableif   enableif

        C++ templates cannot be specialized, so it can be difficult to define specific behavior for certain types. A solution is to use enable_if, which is standard since C++11. 

        template <bool, typename T = void>
struct enable_if
{};

template <typename T>
struct enable_if<true, T> {
  typedef T type;
};

template <typename T>
void do_stuff(typename enable_if<std::is_integral<T>::value, T>::type &t) {
  // an implementation for integral types (int, char, unsigned, etc.)
}

template <typename T>
void do_stuff(typename enable_if<std::is_class<T>::value, T>::type &t) {
  // an implementation for class types
}

      // Foo.h

      Template parameters can be other templates, as in template <template <class A, class B> class TemplateClass>: this is a template which accepts a template class with two template parameters. }

  8. Theory   theory
    1. Action   action

      In Haskell all functions are pure. The term action is used to denote all parts of a program which are not pure, such as IO. Haskell uses its type system to abstract away those parts (note, for instance, that getLine has type IO String and not IO -> String), effectively managing to have pure functions which work with impure values.

    2. Category Theory   category

      The intuition is that Haskell types form a category, where morphisms are all Haskell functions between types. From this point of view, functors are endofunctors, monads are monads and so on. Note however that Haskell does not enforce category theory laws: for instance, the basic properties of functors are not enforced by definition, though it is expected that they should hold. This correspondence is not perfect mainly due to laziness and non-total computations.

      1. Further intuition

        A source of confusion is that a function with signature a -> b can be seen both as a morphism of the category and as an object with type a -> b (remember that -> is just an infix type operator). To make the example with functors, say Maybe, let f be a function with the proper type, Just f is an object with type Maybe (a -> b), so Haskell implicitly treats f as an object, while fmap f is (in this particular context) a function from Maybe a to Maybe b, so Haskell treats it as a morphism.

    3. Constructor   constructor

      Haskell has data constructors and type constructors. The former are functions accepting zero or more values and returning a new value, the latter are functions accepting zero or more types and returning a new type.

    4. Infix   infix

      To use a function as infix, surround the name with backticks, unless the name only contains symbols, in which case it is infix by default.

    5. Polymorphism   polymorphism

      A value is polymorphic if there is more than one type it can have. There are roughly two kinds of polymorphism in Haskell. Note, however, that the same value can present both of them at the same time (they are not mutually exclusive).

      1. Ad-hoc polymorphism

        Similar to the classic sense of polymorphism, in Haskell it is supported with class instances: a type class defines generic functions, then each instance can implement them in a different way. Note that constants can be polymorphic too (for instance, minBound, which has a different meaning if used for Char or Int).

      2. Parametric polymorphism

        The kind of polymorphism of objects whose type contains unconstrained type variables. An example is id, whose type is a -> a, but not (+), whose type is Num a => a -> a -> a (and is an example of ad-hoc polymorphism).

    6. Recursion   recursion
      1. Guarded recursion

        It describes a kind of recursion where the recursive call is hidden under a datatype constructor. Because of lazy evaluation, Haskell can perform all the steps and stop without evaluating the result of the constructor (when not needed). This is why let ones = 1 : ones can be used to produce an infinite sequence of 1's, which can be accessed, for instance, using take (as in take 5 ones). This is also known as productive recursion, because each step of the recursion produces a partial result which can be used. An example of unguarded recursion, instead, is let inf = 1 + inf, which will not work. This kind of recursion gives the possibility to implement the tail recursion modulo cons optimisation, which is a generalization of the tail call optimisation and gives the possibility to implement the recursion without the need of creating a stack frame for each call. foldr is an example of a function which uses guarded recursion, but because of Haskell's laziness, tail recursion modulo cons is not implemented.

      2. Tail recursion

        A recursive function is tail recursive if the final result of the recursive call is the final result of the function itself. If the result of the recursive call must be further processed (for instance, by adding 1 to it), then it is not tail recursive. foldl and foldl' are examples of functions which use tail recursion.

        1. Tail call elimination

          Also known as tail call optimisation, it is the process of optimising tail recursive functions by implementing the recursion as a sort of loop instead of a function call: since no additional processing of the recursive call must be made, there is no reason to allocate a new stack frame and the current one can be used. Note that Haskell has a complicated way of dealing with recursion because of laziness, so tail call elimination is not necessarily implemented: foldl does not use it, while foldl' more or less does.

    7. Symbols   symbol

      Haskell considers some characters as symbols (it should be anything in uniSymbol, since Haskell uses Unicode by default). Among them, there are special symbols (such as parenthesis, comma and so on) which, I think, cannot be used in names. The rest can be used, but only for infix operators. Note however that an infix operator can always be used as a normal function if surrounded by parenthesis. Do not forget that it is not possible to mix symbols and letters in a function definition.

  9. Useful functions
    1. fold   fold

      Folding is the same as reduce, it produces a single value from a list. A good reference is here.

      1. foldr
        foldr f z []     = z 
foldr f z (x:xs) = f x (foldr f z xs)

        This version of fold first processes the first value of the list, then if necessary processes the rest. This can be used to work with infinite lists, if the function f is strict only in the first argument. If the function is strict in both arguments (see for example (+)), then before processing it Haskell must evaluate both arguments, which means that for each element x of the list f x will be pushed onto the stack until the computation ends: this may produce a stack overflow error if the list is too big.

      2. foldl
        foldl f z []     = z                  
foldl f z (x:xs) = foldl f (f z x) xs

        This function will immediately call itself with new parameters until it reaches the end of the list. It makes use of tail recursion, so in principle it can be efficiently executed in a loop, but Haskell does not do it, because it is lazy. What happens is that each recursive call creates a redex (which is a reducible expression) and leaves it there until it is needed. When finally all the list has been processed, the redexes are evaluated starting from the last one and going up the chain: this, again, may produce a stack overflow error. Additionally, this function will hang forever on infinite lists, so it is usually the worst choice for a fold. Nonetheless, there are pathological cases where it is the only option that produces a result (normally when the function which is being folded is lazy in its first argument).

      3. foldl'

        Similar definition as foldl, but it enforces the evaluation of redexes on the go. For this reason it is usually the fastest fold, but it does not work with infinite lists and it cannot short-circuit the computation as foldr does. In addition, the evaluation only processes the top-most constructor, so it may not be enough to completely reduce the redex if the accumulator is a very complex object.

    2. replicate   replicate

      replicate n a creates a list containing a n times.

      1. replicateM

        Monadic version of replicate. It is used to perform an action n times, gathering the results in a list.

  10. Useful typeclasses
    1. Foldable   foldable

      A type class that contains types that can be folded. Note that folding can only happen over some sort of monoid.

      1. Definition
        class Foldable t where
  foldMap :: Monoid m => (a -> m) -> t a -> m

        It is enough to only define the foldMap function to get all the other functions for free.

    2. Functor   functor

      A type class that contains types that can be mapped over. 

      class Functor f where  
  fmap :: (a -> b) -> f a -> f b

      Note that what is usually called a functor in Haskell is only the map between objects of the category, which is only half of the story in Category Theory: the map between morphisms corresponds to fmap. This leads to slightly misleading sentences, such as "fmap (replicate 3) is a function that takes a functor f over any type and returns a functor over a list of elements of that type". It would be more correct to say "fmap (replicate 3) is a function that takes the image of a functor f over any type and returns the image of that functor f over a list of elements of that type".

      1. Applicative functor   applicative

        It is a type class which contains type constructors which are functors. It is basically used as a utility to easily "wrap" functors around functions (see pure) and to apply functions inside functors (see <*>). In other words, it allows to see certain data types as values with contexts and to use normal functions on those values while preserving those contexts. Anyway, more precisely, it is a lax monoidal endofunctor with a strength. 

        class (Functor f) => Applicative f where
  pure :: a -> f a
  (<*>) :: f (a -> b) -> f a -> f b

        Note that to use <*>, the first argument must be a function and the second argument must be a value, both wrapped in an applicative functor. If the function takes more than one argument, there must be as many <*> in order to produce a value.

        1. Laws

          Applicative functors are expected, but not enforced, to satisfy the following laws:

          • pure f <*> x = fmap f x
          • pure id <*> v = v
          • pure (.) <*> u <*> v <*> w = u <*> (v <*> w)
          • pure f <*> pure x = pure (f x)
          • u <*> pure y = pure ($ y) <*> u
        2. Monad   monad

          It is a type class which contains applicative functors which can be "composed in a nice way". Monads could also be described as "computation builders". 

          class Monad m where  
  return :: a -> m a  

  (>>=) :: m a -> (a -> m b) -> m b  

  (>>) :: m a -> m b -> m b  
  x >> y = x >>= \_ -> y  

  fail :: String -> m a  
  fail msg = error msg

          Monads are actually special cases of applicative functors, but there is no constraint because applicative functors came after in the evolution of Haskell. Note that return is exactly the same as pure (see applicative functors). This is a good source to better understand monads.

          1. >>

            The >> function is defined by default and usually it is not reimplemented. Its use is to output a value within a context (the y of the definition), unless its first argument is the zero of the monad, in which case the output will be the zero itself. The difference with >>= is that it is applied to a value with context instead of a function. There are surprisingly many cases when this is useful, for instance if used in combination with a guard function which outputs a minimal successful or unsuccesful value (in the case of Maybe, it could be either Just () or Nothing) depending on the result of a boolean computation. Used in a chain of >>=, its purpose is to abort the computation based on the result of some evaluation which we are not interested in the result of, apart from this success/fail condition.

          2. do

            The do notation is synctactic sugar used to replace a chain of nested >>= and >>. The lines which contain <- correspond to >>= while the others correspond to >>. The return at the end is only used to wrap the result in a suitable context: it does NOT return a value in the classical sense!

          3. Laws

            Monads are expected, but not enforced, to satisfy the following laws:

            • return x >>= f = f x
            • m >>= return = m
            • (m >>= f) >>= g = m >>= (\x -> f x >>= g) (associativity)

            Basically the first two laws say that the unit is a neutral element, while the third is associativity.

          4. List   list

            When lists are used as monads, they represent non-determinism: the value of something is not fixed, but is taken from a list of possible values. The do notation, in particular, is equivalent to list comprehension (which is actually just synctactic sugar for the do notation).

          5. Maybe   maybe

            This monad is used to represent computations that may not yield a result. It is similar to using null in object-oriented languages, but has some advantages. One of them is that there is no need to check for Nothing at every step of the computation, cause a Nothing will be propagated down the chain and one check at the end is enough.

        3. Useful related functions
          1. <$>

            Utility function used to quickly apply functions inside functors. No need to have the function already wrapped in a functor, as it would be the case if only using <*>. The definition is 

            (<$>) :: (Functor f) => (a -> b) -> f a -> f b
f <$> x = fmap f x

            where of course the f in the first line and the f in the second are not related. Usage is, for instance, 

            (+) <$> Just 1 <*> Just 2

            which produces Just 3. Note that this syntax is equivalent to 

            pure (+) <*> Just 1 <*> Just 2
          2. liftA

            Same as fmap but for applicative functors only. There are also versions called liftAx, where x is a number (I think up to 3). They work the same but accept functions with x arguments (whereas fmap accepts functions with only one argument). Of course the number of applicative functors provided as arguments must change accordingly. All in all, they are the same as 

            f <$> a arg1 <*> a arg2 <*> a arg3 ... a argn.

            where f is a function which accepts n arguments and a is an applicative functor.

          3. sequenceA (for lists)
            sequenceA [] = pure []
sequenceA (x:xs) = (:) <$> x <*> sequenceA xs

            Utility function used to transform a list of functors in a functor of a list. Actually, the definition of the Prelude works with any Traversable, not only lists. If used on a list of functions, it can be used to apply all functions to the same argument and put the results in a list. If used on a list of lists, it forms all combinations of elements. If used on a list of IO actions, it performs all of them in order and returns the results in a list. Because of this behaviour, it is said that sequenceA is used to evaluate "non-deterministic" computations. Note that the apparent variety of behaviours depends on the implementation of <*> provided by the different applicative functors.

          4. traverse
            traverse :: Applicative f => (a -> f b) -> t a -> f (t b)
traverse f = sequenceA . fmap f

            Useful to apply a function with fmap to a structure t a and then pack together the results with sequenceA.

            1. Example

              traverse (const ["a","b"]) [1..3] produces all lists of three elements taken from ["a","b"].

      2. map

        Note that map apparently only exists for pedagogical reasons, and it is basically the restriction of fmap on lists.

    3. Monoid   monoid

      A type class that contains types that support associative operations with a unit.

      1. Definition
        class Monoid m where  
  mempty :: m  
  mappend :: m -> m -> m  
  mconcat :: [m] -> m  
  mconcat = foldr mappend mempty

        Note that actual monoids only have to implement mempty and mappend.

        1. Laws

          Monoids are expected, but not enforced, to satisfy the following laws:

          • mempty `mappend` x = x
          • x `mappend` mempty = x
          • (x `mappend` y) `mappend` z = x `mappend` (y `mappend` z)

  1. Core language
    1. Abstraction

      It is the process of hiding implementation to focus on features.

    2. Anonymous inner classes   anonymousinnerclass class

      Unnamed inner classes that implement an interface. Could use a lambda expression instead if the interface is functional (only one abstract method). Maybe works also when implementing classes (not just interfaces). 

      interface MyInterface {};

    3. Checked and Unchecked Exceptions   checked unchecked exception

      Checked exceptions are checked at compile-time: any method that throws them must either catch them or use a throws declaration in its signature. Unchecked exceptions are not checked at compile-time, so their effect is only visible at run-time (they are direct subclasses of RuntimeException). The idea is that checked exceptions should be used in a method when we expect the client code that calls the method to do something about a specific error: using a checked exception has the effect of notifying the client code of the possibility of the error and to force it to take countermeasures. Unchecked exceptions, on the other hand, are the result of programming errors or bad input: client code should still handle them, but they are not checked at compile-time in order to reduce the "visual" complexity of the code.

    4. Classloader   classloader

      A classloader is a Java class whose purpose is to load other Java classes. Some classloaders are defined by the JVM, others by the application container and others by libraries or users. Classloaders form a rooted tree, each one referencing its parent.

      1. Classloader leaks   leak

        Classloaders hold references to all the classes they loaded, which in particular contain static fields. This means that as long as even a single object is alive, its classloader will stay alive, potentially using up a lot of memory.

        In a Java EE application, where a JVM runs on a server, it is not unlikely that some objects survive redeploys, thus leading to memory leaks and OutOfMemoryError exceptions.

      2. Class identification   identification

        Classloaders are also used together with fully qualified names in order to uniquely identify a class. The fully qualified name is not enough, otherwise an application could shadow core Java classes using custom definitions.

        As a side effect, the same class loaded from different classloaders will look different from the point of view of the JVM. A common reason for this happening is when different versions of the same library are bundled in different parts of the application.

      3. Details

        Due to the lazy nature of Java, classes are loaded at runtime when necessary. This is the purpose of classloaders. When a JVM is started, some platform specific machine code loads the first core classes: this is known as the bootstrap classloader (though the name is misleading as this is not a true classloader!).

        The bootstrap classloader also loads the first classloader of the tree, the extension classloader, whose purpose is to load all .jar files kept in the java.ext.dirs path. Then, the system classpath classloader is loaded, which loads classes in directories and jar files specified by the CLASSPATH environment variable, the java.class.path system property or the -classpath command line option.

        1. Delegation

          The tree of classloaders usually works by delegation: each classloader leaves the task of finding a class to the parents and only if they all fail it tries to locate it. In some context (for instance, in some Java EE application), the behaviour is reversed, though this can lead to numerous problems.

      4. Related exceptions   exception
        1. NoClassDefFoundError

          Thrown if the JVM or a classLoader instance tries to load in the definition of a class and no definition of the class can be found. Even if the project was successfully compiled it may happen that some resources are not available at runtime.

          A possible solution is to manually examine the classpath of the various classloaders.

        2. NoSuchMethodError

          Thrown if a class does not contain a given method. This usually means that the wrong version of the class was loaded.

    5. Concurrency
      1. Synchronization

        When a thread calls a synchronized method or enters a synchronized block, the object is locked and no other thread can access synchronized methods in that object (in case a thread tries, it enters a waiting state). Synchronization is necessary to regulate access to shared resources and to prevent race conditions.

      2. Monitor

        It is the Java implementation of a mutex (or lock), together with condition variables (associated to waiting threads). Every object has a monitor, which prevents concurrent access if any of the thread accessing the object requires it.

      3. wait

        If a thread running in a synchronized environment is stuck, it can call obj.wait() to release the object while it waits. obj.notify() o obj.notifyAll() ne faranno riprendere l'esecuzione quando chiamati da un altro thread.

    6. Dynamic classloading

      All Java code is associated with methods contained in classes. Simplifying, when a class is loaded (which includes when, but is not limited to, an object of that class is being instantiated), a class with all the methods is loaded into memory. When a non-static method is called, a reference to the class is obtained (every object keeps such a reference) and the method is called passing an implicit this argument (this looks similar to Python). See also Class identification for additional details about how a class is identified.

    7. Functional interfaces

      Any interface that conatains exactly one abstract method (non-abstract and static methods do not count). This method can be implemented using lambda expressions. 

      interface MyFunctionalInterface {
    int sum(int n, int m);
}
// then, in a method
MyFunctionalInterface f = (n,m) -> n+m;
    8. Generic functional interfaces

      A functional interface can be generic. The result is that a lambda expression will accept generic arguments, inferred by the types with which the interface is instantiated.

    9. Inheritance

      A class can inherit from only one class. The constructor must call super(...) as first thing. If it does not, super() is implicitly added. Private members are not inherited. If Y extends X, X=Y is admissible but X can access only the part of Y inherited by X (note however that if one tries to access an overriden method it accesses the new one, which is known as virtual method invocation). The idea is that the JVM when it creates an object first creates all its superclasses. Inherited methods can be overriden or hidden (the latter if they are static).

      1. Annotations   annotation
        1. @Cascade   cascade

          This property annotation is used to set cascade behavior on @OneToMany and @OneToOne (maybe also @ManyToOne?) properties. The argument is a org.hibernate.annotations.CascadeType constant. The tag can be coupled with the cascade attribute of @.To. tag. Doclink.

        2. @DynamicInsert   dynamicinsert

          This class annotation is used to tell hibernate to only include initialized properties in an insert query. It can be useful if, for instance, there is a trigger on the DB that should be fired on insert if a field has a null value, but the field is initialized by Java to a default non-null value. A similar (but more extreme) effect should be achievable field by field with the insertable = false attribute of the @Column tag.

        3. @IdClass   idclass

          This class annotation is used to implement composite IDs. The use is @IdClass(IdClass.class), where IdClass is a Serializable external class that works as identifier: it must contain the same properties as the identifier properties (the ones that have the tag @Id) with the same name and type (unless the identifier property is an association @OneToOne or @ManyToOne, in which case it should have the type of its primary key). It must also contain a constructor which initializes all the properties (and a default empty constructor) and the properties should be protected. Hibernate will then use the equals method of IdClass (which must be overriden) to check for equality. NOTE: It looks like this approach is old and somehow deprecated.

      2. Basic theory   theory
        1. Architecture   architecture

          Hibernate provides a layer between low-level Java APIs such as JTA and JDBC, and the rest of the application, thus encapsulating the persistence logic.

        2. Configuration file   configuration file

          Natively, Hibernate uses a configuration file called hibernate.properties or hibernate.cfg.xml. However, since it is JPA-compliant, it is also possible to use a persistence.xml file located in the META-INF folder.

        3. Configuration object   configuration

          Created only once during application startup, parsing configuration files. It provides two key components, the database connection and the class mapping setup.

        4. SessionFactory object   sessionfactory

          Thread-safe, heavyweight object, usually created only once during application startup. Each database the application connects to needs its own SessionFactory. The JPA equivalent is the EntityManagerFactory, and Hibernate basically uses the same implementation for both. The SessionFactory maintains services used by all Session objects, such as second level caches, connection pools, and so on.

        5. Session object   session

          Used to get a physical connection with a database. It is lightweight and designed to be instantiated each time an interaction is needed. Usually it is not thread-safe, and it should not be kept open for too long. It models the concept of a unit of work. The JPA equivalent is the EntityManager. Behind the scenes, it wraps a JDBC java.sql.Connection and acts as a factory for org.hibernate.Transaction.

        6. Transaction object   transaction

          Single-threaded, short-lived object used by the application to demarcate individual physical transaction boundaries. The JPA equivalent is the EntityTransaction. Both are abstractions from the underlying transaction system in use (JDBC or JTA).

        7. Domain model   domainmodel

          The set of persistent classes is usually called domain model, as it describes the problem domain of the application. It is the central character in an ORM.

          1. Type   type

            The ability to read/write data from/to the database is the function of a Hibernate type.

            1. Value type   value

              A piece of data which does not define its own lifecycle: it is owned by an entity. It is also called a persistent attribute.

              1. Basic   basic

                A basic type usually maps a single database column to a single non-aggregated Java type. Hibernate provides a number of built-in basic types. Reference.

              2. Embeddable   embeddable

                An embeddable type, also known as component in Hibernate terminology, is a composition of values. It can be made up of basic values as well as associations, with the caveat that, when used as a collection element, it cannot define collections itself.

              3. Collection   collection

                A collection of value types, embeddable types or entities. Objects in the collection may be handled with value semantics (lifecycle depends on the collection owner, which is always an entity, event if the collection is contained in an embeddable) or it may be a reference to another entity with its own lifecycle. Hibernate uses its own implementations of collections (which are enriched with lazy-loading, caching or state change detection semantics), so it is important to use interfaces to define the types (such as List instead of ArrayList).

            2. Entity type   entity persistent

              Entities exist independently of other objects, they are domain model classes (also called persistent classes) which correlate to rows in a database table, using a unique identifier. There are some rules all persistent classes should follow, though it is not a hard requirement.

              • Be POJOs,
              • Have a default constructor,
              • Contain an ID,
              • Have private attributes and getters and setters with usual naming convention,
              • Be non-final, or implement a public interface (needed in order to use Hibernate proxies).
              1. Naming strategy   naming

                Part of the mapping of an object model is mapping names from the model to the database. Hibernate does this in two steps (the logical name and then the physical name): both can be customized, the latter giving more freedom. JPA defines only one naming strategy though, which is implemented in ImplicitNamingStrategyJpaCompliantImpl.

      3. Cache   cache

        There are three flavors of caching available: session, second level, query.

        1. Session   session

          Caches objects within the current session. It is enabled by default. It is also called first-level cache.

        2. Second level   secondlevel

          Caches objects across sessions. It works when objects are loaded by primary key, which includes fetching associations. By default, it is not enabled.

          1. Concurrency strategy

            A mediator responsible for storing items of data in the cache and retrieving them from the cache. When using the second-level cache, it is necessary to specify the concurrency strategy for each persistent class and collection.

            Strategy Description
            Transactional Read-mostly data where it is critical to prevent stale data in concurrent transactions.
            Read-write Apparently, the same as transactional.
            Nonstrict-read-write No guarantee of consistency, use for data that does not change often and which is not critical to keep up-to-date.
            Read-only Suitable for data that never changes.
          2. Cache provider

            Interface implemented by objects which provide second-level cache. There can be only a single cache provider for the whole applications. There is a number of standard ones, otherwise a new one can be implemented. Note that the standard ones are not compatible with all the concurrency strategies. Second-level caching may downgrade the performance of the application: it should only be used after profiling.

        3. Query   query

          Caches the result of queries, for each combination of parameters they are executed with. To activate it, set to true the hibernate.cache.use_query_cache property. Then, for each query, use setCacheable and setCacheRegion.

}

@Entity
@Table
public class Child {    
    @Id
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)

}

      8. Working with objects   state

        Hibernate defines and supports the following object states:

        State Description
        Transient An entity which has just been instantied with the new operator. It is not associated with the Hibernate session.
        Persistent An entity which has a representation in the database and an identifier value. It is in the scope of the Hibernate session.
        Detached An entity which was persistent but whose session has been closed. The reference is valid, and it can be reattached to a new session.
        1. Insert   insert

          Use save or persist.

          1. save   save

            Guarantees to return an identifier. If an INSERT has to be executed to get the identifier, it happens immediately, even if outside of a transaction.

          2. persist   persist

            It does not guarantee that an identifier will be assigned to the persistent instance immediately, it may happen at flush time. It guarantees that it does not execute an insert outside of a transaction though.

        2. Select   select

          Use load or get.

          1. load   load

            It is a sort of lazy loading. It does not necessarily hit the database immediately, but when it does if it does not find a match it will throw an error.

          2. get

            Used to load an object when it is not sure that it exists. Note that it immediately hits the database. It is also possible to specify a lock mode.

        3. Update   update

          Use update or merge.

          1. update   update

            It persists and reattaches the given detached object. It throws an exception if another object with the same ID is found in the session.

          2. merge   merge

            It copies the state of the given object to the state of a corresponding object in the session (where corresponding means with the same ID). It does not attach the given object, but returns the previously attached one. Note that Hibernate synchronizes the state of the entities with the state of the DB when the session is flushed, so calling merge is enough to update an object. Subsequent changes to the merged entity will not be persisted though (unless merge is called again).

      Dependency injection is the way Angular provides testable components.       myMod.config(function($provide) {
    $provide.provider('greeting', function() {
        this.$get = function() {
            return function(name) {
                alert("Hello, " + name);
            };
        };
    });
});

      After creation, a service can be injected in any injectable function (like a controller): Angular will call the provider's $get function to obtain a new instance of the service (see here for details). Following the previous example: 

      myMod.controller('MainController', function($scope, greeting) {
    $scope.onClick = function() {
        greeting('Ford Prefect');
    };
});

      Notice that factory, service, and value are just shortcuts that can be used to do the exact same work that can be done using provider. Under the hood, there is absolutely no difference.       2. Injector

  7. Theory   theory
    1. Anonymous function   function

      A function witout a name. A function declaration cannot be anonymous, but a function expression can. The latter example is useful when modularising scripts, as the content of the anonymous function is private and cannot produce conflicts.

    2. call, apply, bind   function

      call is used to execute a function using a different context (that is, the value of the variable this). Recall that the value of the context is usually determined by the scope of the function definition. Use: myFunc.call(myContext, argument1, argument2, ~...~). apply is similar to call, except that it takes an array-like object instead of listing the arguments. Use: myFunc.apply(myContext, array_of_arguments). bind is used to bind the context and/or one or more arguments. It returns a value of type function for later use. Use: var bindedFunc = myFunc.bind(myContext, arg1, arg2, ~...~).

    3. Expression   expression

      An expression is a piece of code that produces a value. It can be written wherever a value is expected (such as an argument of a function call). Example: myVar, 3 + x, myFunc(a, "b").

      To prevent ambiguity, an expression statement cannot start with { or function. If needed, just surround it using ().

      1. Object literals and blocks

        It is an expression which produces an object. Example: { foo: bar(3,5) } or simply {} Note that it cannot be distinguished from a statement (a block is a statement, every statement can be prefixed with a label such as foo:, and bar(3,5) can be seen as an expression statement). This is why {}+[] and []+{} yield different results (in the former, {} is seen as a statement).

      2. Function expressions and function declarations

        function () {} is a function expression (which returns an anonymous function as value) and it can also be written as function foo() {} (where foo is a function name that only exists inside the function itself, for instance to be used for recursion).     7. Prototype   prototype

      Every object has a prototype (which itself is an object though in some cases may be null), which is stored in an internal property often called [ [ prototype ] ]. The Object.getPrototypeOf method is a getter for this property (in some systems there may be a __proto__ property which contains it). Every function has a prototype property which is initialized as a nearly empty object (and can be extended). When a new instance of a function is called, using it as a constructor, the prototype of the new object will point to the constructor's prototype property and therefore it will be different from the function prototype, which is Function.prototype. On the other hand, if an object is created using Object.create, then it inherits directly from the one given as first argument: that is, its prototype is exactly the object given as argument.

      1. Prototypal inheritance

        It is simply a terminology used to describe the way Javascript inheritance works. In addition to these considerations, it should be noted that when a property is not found in an object, Javascript automatically looks for it in its prototype, going all the way up to the root. On the other hand, if a property with the same name is defined in the child object, it shadows all other properties up the chain. It is important to consider that properties are just references to objects, so this creates an apparent asymmetry between "primitive" and "complex" objects: for instance, modifying a string in the prototype will fail (in fact, the modification will be an assignment, which creates a new property in the child and shadows the old one).

      A package is just a directory tree with a __init__.py file at its root. It works as a namespace (which is basically a dictionary of variable names and their corresponding objects).       1. import   import

        The import statement is used to load a module from its name. It makes the name available in the current scope (so this means there will be a variable which contains a reference to the module. Note that the variable can be used to modify module attributes). The Python interpreter first looks for a module in the cache (so importing more than once has no negative effect), then, in order, in the directories specified by the environment variable PYTHONPATH, in the default system directory (in Linux, /usr/local/lib/python), and in the current (working) directory (not sure about this order but it makes sense!).           The from ... import ... can be used to load only specific attributes to the current scope. The first argument can be a module or a package name, or also package_name.module_name. Note that the whole module is still loaded in the cache, so there is no performance advantage over a normal import. There is also the version from ... import *, which introduces all attributes (use sparingly).

          1. Relative import   relative

            The from import syntax can be prefixed by a relative path: for instance, from .module_name import attribute can be used to load an attribute from a module in the current directory.   2. Coroutine   coroutine

    Essentially, it is a function whose execution can be paused. Coroutines are strictly linked to the notion of event loop, which is a programming constructs that waits for and dispatches events in a program and is used to manage asynchronous programming.

    A coroutine can be defined with the async def construct and can pause execution using await on an awaitable object (which can be another couroutine, or more generally an object that defines an __await__ method which returns an iterator which is not a coroutine). Note that generators behave similarly, but they are not related to an event loop.

    Before python 3.5, a couroutine was any method annotated with @asyncio.coroutine and it used the yield from construct to pause execution (so any generator with that annotation was automatically a coroutine). The yield from would return an asyncio.Future object that the event loop monitored to know when to resume the coroutine execution (actually this is true only if asyncio is used as event loop, which is not required). These coroutines are now known as generator-based coroutines. The main difference between the old and new syntax is that yield from accepts generators while await does not, but it accepts awaitable objects. They both accept coroutines though. Another difference is that await cannot force an object to be passed down to the event loop (whatever that means…).

    This is basically a convenient way of implementing the decorator pattern. The decorator acts as a wrapper that adds functionality before and after a function call. 1.12.8 Theory   theory

  1. APIs and frameworks   api framework

    A framework is a group of classes, interfaces and other pre-compiled code upon which an application can be built. An API is the public face of a framework.

  2. Asynchronous programming   asynchronous

    Often, a program has access to plenty of CPU time, but its speed is limited by some slow operations (typically, I/O operations). This is the reason that brought to the development of asynchronous programmig. Originally, this term included every style of non-synchronous programming, but it is now used to describe a style that uses a single thread of execution and is divided in a series of tasks (as much independent as possible) that can be interleaved.

    1. Advantages over multi-threaded programming

      It is the programmer that chooses when tasks should be supended (as opposed to the operating system) and two tasks are guaranteed not to run at the same time (thus preventing memory contention issues). This makes implementing complex systems a little easier. Note that however there can always be race conditions, as the order in which tasks are executed is not up to the programmer.

    2. Advantages over synchronous programming

      Programs can be speeded up quite a bit when there is a large number of largely independent tasks performing blocking operations. The typical situation is that of a network server.

    3. Drawbacks

      Since a task is never interrupted, there can be functions that starve the rest of the program, thus killing the whole purpose of asynchronous programming.

  3. Closure   python javascript closure

    A closure is a function object that remembers values in enclosing scopes regardless of whether those scopes are still present in memory. It happens in languages that permit the return of a function and that give to nested functions access to the enclosing scopes (see Python and Javascript for example). The idea is that the enclosing scope can be used as an immutable state storage, even though it is not necessarily immutable. Mathematically speaking, this is basically a way to implement partial function application (such as f_x(y) = f(x,y)).

  4. Design By Contract   designbycontract

    Approach for designing software which prescribes that every component should define a formal, precise and verifiable interface, known as a contract. In particular this contract must specify preconditions, postconditions, invariants, side effects and error and exception condition values. The concept was coined by Bertrand Meyer when designing the Eiffel language (1986).

  5. Difference between Expression and Statement   expression statement

    In most languages, statements contrast with expressions in that statements do not return results and are executed solely for their side effects, while expressions always return a result and often do not have side effects at all. In languages that mix imperative and functional styles, such as the Lisp family, the distinction between expressions and statements is not made: even expressions executed in sequential contexts solely for their side effects and whose return values are not used are considered "expressions". In purely functional programming, there are no statements: everything is an expression. A distinction is frequently observed in wording: a statement is executed, while an expression is evaluated.

  6. Difference between Programming Paradigm and Design Pattern   paradigm pattern

    A programming paradigm is a way of classifying a programming language according to the style of computer programming (such as imperative, object-oriented, functional and so on). A design pattern is a general reusable solution to a commonly occurring problem within a given context in software design.

  7. Expression   expression

    It is a combination of one or more explicit values (expressions that cannot be evaluated any further), constants (values that cannot be modified), variables, operators, and functions, that can be evaluated to return another expression. In functional programming, the resulting expressions are often functions, which can themselves be further evaluated.

  8. Paradigms   paradigm
    1. Aspect-oriented programming   aspect

      A paradigm that aims to create modularity by allowing the separation of cross-cutting concerns, which is a complicated way of denoting all the "side-effects" that affect multiple parts of an application: the prime example is logging, while other examples may be security or caching.

      1. Aspect

        A module which has a set of APIs providing cross-cutting requirements. An application can have any number of aspects.

      2. Join point

        A point in an application where AOP aspects can be plugged-in.

      3. Advice

        The actual action to be taken.

      4. Pointcut

        A set of one or more join points where an advice should be executed.

      5. Weaving

        The process of linking aspects with objects to create an advised object. This can be done at compile time, load time or runtime.

    2. Declarative programming   declarative

      A paradigm in which programmers specify the wished result, without enforcing a specific resolution strategy. I think this could be somehow related to the notion of axiomatic semantics. An example of this style of programming is SQL. This can lead to very easy implementations of difficult algorithms, but there will always be cases in which the programmer needs to manually tune the algorithms which are to be used (in SQL, this is the case when database hints are used).

    3. Functional programming   functional

      A paradigm that treats computation as the evaluation of mathematical functions and avoids changing-state and mutable data. It is a subset of the declarative programming paradigm. It has its origins in lambda calculus.

      1. Currying   currying

        It is the technique of translating the evaluation of a function that takes multiple arguments into evaluating a sequence of functions, each with a single argument.

      2. Higher-order function

        A function that takes as one of its arguments and/or returns a function as its result.

      3. Referential transparency

        An expression is referentially transparent if its value only depends on the values of its subexpressions, or, in other words, it only depends on its textual context and not on the computation history. In mathematics, this is true for all functions, but in programming this is true only if functions do not have side effects (and are therefore deterministic). Referential transparency is considered one of the cornerstones of functional programming.

    4. Object-oriented programming   object oriented

      A paradigm in which a program is composed of discrete units called objects, each of which functions to some extent like a program in its own right. It is an evolution of structured programming, but it emphasizes data over action and separtion of data and procedures.

      1. Data access object (DAO)

        An object that provides an abstract interface to a persistence mechanism (such as a database). It maps application calls to the persistence layer, providing some specific data operations without exposing the details.

      2. Encapsulation

        Also termed data hiding, it is the possibility of protecting an object internal data, i.e. state, from external modifications. Only publicly available methods can be used to modify the state.

      3. Inheritance

        Possibility of an object to inherit some functionalities from another. It makes code reuse easier.

      4. Object composition

        It is an alternative to class inheritance. Required functionalities are implemented separately following well-defined interfaces, then higher-level objects are constructed assembling them. The main advantage is that specific implementations can be changed more easily, even at run-time. The main drawback is that it often generates more redundant code, as all methods must be rewritten for each new implementation (as opposed to inheritance, which only requires to rewrite overriden methods). This can be mitigated by the application of the delegation pattern.

      5. Polymorphism

        It is the possibility of not choosing a specific implementation until run-time (also known as dynamic binding), and of changing implementation without changing the rest of the logic. Since an object, externally, is equal to its interface, a client object can decide to support an interface without forcing a specific implementation. Languages supporting polymorphism assign the same type to objects implementing the same interface (which may be seen as a more general case of inheritance), thus allowing this freedom.

      6. Reflection

        It is the ability of a program to examine and modify its structure and behaviour at runtime. In Java it is implemented, for instance, for the Class<T> class.

      7. Serialization

        Mechanism using which an object can be converted to a sequence of bytes and back, for instance for transmission. It is platform-independent.

      8. SOLID

        Acronym used to denote the five basic principles of object-oriented programming.

        1. Single responsibility principle

          A class should have only a single responsibility, that is it should only have one reason to change.

        2. Open/Closed principle

          Software entities (classes, modules, functions…) should be open for extension but closed for modification. This principle is closely related to inheritance and polymorphism. At the basic level, it requires that every implementation adheres to a contract that will never change. This is required in order to eliminate the need to change every other part of the code that makes use of the implementation. One way to implement this is to extend a class whose functionality must be enhanced.

        3. Liskov substitution principle

          Objects in a program should be replaceable with instances of their subtypes without altering the correctness of that program. This means that a child object should never distort the functionalities of the parent, so that code which correctly works with the parent will still correctly work with the child.

        4. Interface segregation principle

          Many client-specific interfaces are better than one general-purpose interface. The idea is that a client using an interface should only have access to the relevant methods. This reduces the coupling and the noise in general. This principle can be implemented creating client-specific interfaces which only expose parts of a general-purpose object.

        5. Dependency inversion principle

          One should “depend upon abstractions, not concretions.” This means that interactions between modules should be moderated by abstractions, and that implementation details should be influenced by the chosen abstraction and not the opposite. Again, this is done to reduce coupling.

    5. Structured programming   structured

      Based on reusable, discrete code entities and data scope. It emphasizes the importance of one-in one-out control constructs, that is, constructs that have one point of entrance and one of exit (no wild goto statements). The flow of the program is strictly fixed: only sequences, selections and iterations are admitted. In short, it is the style of programming of Informatica 1 at Tor Vergata.

    6. Symbolic programming   symbolic

      A paradigm in which the program can manipulate its own formulas and program components as if they were plain data. Examples of languages which support this are LISP and Wolfram Language. This approach makes complex data structures very easy to create.

  9. Patterns   pattern
    1. Functional   functional
      1. Monad   monad

        Design pattern used to describe computations as a series of steps. It is extensively used in pure functional programming languages to manage side effects. Monads are used to wrap types giving them additional behaviour. The reference for this description is here.

        1. Components

          To be considered a monad, a structure has to provide three components. It is also required to satisfy the same laws that hold for Haskell's monads.

          1. Type constructor

            A feature that creates a monadic type given an underlying type.

          2. Unit

            A function that wraps a value of underlying type into a monad. This is the same as return for Haskell's monads.

          3. Bind

            A function that chains operations on monadic values. This is the same as >>= for Haskell's monads.

        2. Monads in object-oriented languages   object oriented

          Monads can be implemented in multiparadigm languages, provided that they support some functional features. In this case, the unit function could be represented using a constructor and the bind function using an instance method. 

          interface MStatic<T> {
    // constructor that wraps value
    new(value: T): M<T>;
}

interface M<T> {
    // bind as an instance method
    bind<U>(transform: (value: T) => M<U>): M<U>;
}
    2. Object-oriented   object oriented
      1. Bridge   bridge
        1. Motivation

          This pattern is used to decouple an abstraction (interface) from its implementations. It is useful when we want to make them as independent as possible, for instance because implementations should be changed at run-time.

        2. Implementation

          An interface Abstraction is accessible from the client and can be freely subclassed. A second interface, Implementor, is used from concrete implementations: it can be quite different from Abstraction and can vary independently (usually it contains low-level methods, while Abstraction contains higher-level ones). Abstraction then just forwards to Implementor the requests it gets from the client (their relation is called a bridge). To choose a concrete implementation at run-time, it is often useful to use an Abstract Factory.

        3. Consequences

          This pattern creates a middle layer that shields clients from implementation details and let both parties evolve separately.

      2. Command   command
        1. Motivation

          This pattern is used to encapsulate a request as an object, in order to parameterize different requests and implement queues, logs and undo-able operations.

        2. Implementation

          A generic Command interface is created to model a command (possibly just definining a single execute method). Concrete implementations of command are also provided upon creation with the receivers of such commands (the objects that know how to perform the specific command), thus being independent from the invokers, which are the objects that host, and activate, them (such as, for instance, a graphical menu button). Concrete commands may also be macro commands, which call a series of primitive commands in a sequence (this is an instance of the Composite pattern). To support undo operations, the command may store the information needed to reverse its effect and register itself in the list of performed operations when called. Similarly, it can support logging.

        3. Consequences

          How much a command should do and how much should be left for the receiver depends on many factors and can vary from the extreme of a command that does all the work to the extreme of a command that just forwards the call.

      3. Chain of Responsibility   chain responsibility
        1. Motivation

          This pattern is used when there are requests that should be handled dynamically.

        2. Implementation

          All candidate handlers implement the same interface (or extend the same class) which contains a handleRequest(...) method that either handles the request or forwards to the next handler in the chain (according to the type of request). The next handler may be contained as a reference inside each handler object (or maybe there could be a parent class containing lists?). Requests can be built-in objects, codes, or any other possible implementation, as long as they contain all the context needed to handle them.

        3. Consequences

          One possible risk is that a request is not guaranteed to be handled.

      4. Composite   composite
        1. Motivation

          This pattern is used when there are both primitive objects and composite objects (which form together a tree-like structure), but we do not want to make a distinction between them (because we do not want clients to implement a different set of behaviours according to the type). It is often applied in UI design.

        2. Implementation

          The pattern is usually implemented defining a common inteface for both primitive and composite objects (called component), in addition to separate interfaces for them (called leaf and composite). If primitive methods are called on leaves they directly execute it, if called on composites they usually forward the call to their children (in addition to applying some logic if necessary). As such, objects should always know their children (and often their parent too). One important choice is whether to put children manipulation methods in the component interface or in the composite one. The trade-off is between transparency and safety. To ensure type safety if transparency is chosen, it is possible to define a getComposite() method in the component interface, which will return null for leaves and a reference for composites. Another approach (to avoid type testing) is to use default implementations (which raise exceptions) of composite methods.

        3. Consequences

          One disadvantage is that it can make the code too general, i.e. it gives no way of distinguishing between different types of primitive or composite objects.

      5. Decorator   decorator
        1. Motivation

          This pattern is used to add temporary responsibilities (or functionalities) dynamically to objects, or more generally when subclassing is impractical.

        2. Implementation

          Usually there is a common interface (called component), from which both a ConcreteComponent class and a Decorator interface inherit. The decorator maintains a reference to a component object to which it forwards all methods call, after "decorating" it (i.e. after applying some logic to it). Applying decorators to an object is transparent, in the sense that the client cannot (and does not need to) distinguish between working with a concrete object or a decorated one.

        3. Consequences

          Since all decorators implement the component interface and since there may be a lot, that interface should be kept as lightweight as possibile. When this is not practical, alternatives should be used, such as the Strategy pattern (which lets the guts change instead of the skin).

      6. Delegation   delegation

        Approach that handles requests using two layers: a receiver object receives the request and delegates it to a delegate object that processes it (which may be an object that the receiver holds a reference of). Note that in the "complete" version of delegation, the receiver passes a reference to the delegate in order to let it know who called it (languages which support this pattern have this behaviour by default). This gives great flexibility and is often used together with object composition to replace and enhance class inheritance. The drawback is that the program becomes less easy to read and also less efficient.

      7. MVC (Model View Controller)   mvc
        • Model: the core of the application, implements the business logic. It is the layer that stores (and retrieves) data.
        • View: the external layer of the application, the only one a final user can interact with. It describes how data should be displayed.
        • Controller: the middle layer of the application, processes user actions using an event-driven approach. It is usually dependent on the View, even if it would be better to design it as independent as possible. It is the layer that modifies data (to enable communication between model and view).
      8. Observer   observer

        Used to update a set of dependent object upon the change of a subject object. It poses no limits on the number and types of the dependent objects, they only have to register themselves and implement some generic interface to retrieve the updates.

        • Subject: interface for adding and removing observers.
        • Observer: interface for registering and receiving notifications.
      9. Singleton   singleton

        Used when something must be unique. In Java, for instance, if this something is a class one can use a private constructor.

      10. Strategy   strategy
        1. Motivation

          This pattern is used to encapsulate a family of algorithms and make them interchangeable.

        2. Implementation

          We can use an interface Context as a client handle to call an algorithm. The Context will contain a Strategy object, where Strategy is an interface common to all algorithm implementations. To pass data from Context to Strategy there are two possibilities: either the data is passed as argument (easier to maintain but a lot of unnecessary data may be passed to less complicated algorithms) or Context defines an interface to access data (more precise but makes the decoupling less effective).

        3. Consequences

          Clients should know about different algorithms and be able to choose one. This can be mitigated providing a default choice.

      11. Visitor   visitor
        1. Motivation

          This pattern is used when we have to implement a varying number of operations that all work on a stable object structure, that is, a number of different classes that are somehow related (for instance, they may implement the same interface), and we do not want to mix these operations with the structure itself.

        2. Implementation

          For each operation there must be a corresponding class which implements a common Visitor interface. This interface contains a method visitConcreteElement(*ConcreteElement elem) for each ConcreteElement in the the object structure. The Element interface common to all ConcreteElement defines an accept(*Visitor v) method, whose implementation just calls the visitConcreteElement(...) method corresponding to the correct ConcreteElement (this is also known as double-dispatch, as the operation which is executed depends both on the type of visitor and the type of element). In this way, the visitor does not need a type cast to know which element is visiting, and can implement its algorithm applying the correct operations to each element; in addition, it can store local state inside class variables. Finally, a consideration that must be done is who is responsible for traversing the elements. It could be the structure itself, an iterator, or the visitor. This last approach is useful for complicated iterations, where the next element depends somehow from the algorithm and the previous elements.

        3. Consequences

          Adding new operations is easy, but adding or modifying elements can be very expensive, as all visitors must be modified. In addition, visitors must use the interface provided by the elements to do their work, so there is the risk of them requiring rich interfaces, which may compromise the incapsulation.

  10. Runtime   runtime

    It is the set of software/instructions that are executed while a program is running, especially the ones that were not contained in the program itself. In other words, it is the code required to implement the features of the language itself.

    Low-level languages like C have very small (if any) runtime, while more complex languages like Java have much more extensive runtime (see the Java virtual machine).

  11. Semantics   semantics

    The term refers to the meaning of languages, as opposed to their form, which is the syntax. The concept of meaning is sort of vague. Basically the idea is that a computer program must somehow eventually produce a computation, which is a physical change in the state of the internal memory of a computer. That computation can be seen as the meaning (of the program), and the important thing is that we ensure that there is no ambiguity in this translation process (note that both semantics and syntax of programming languages can be precisely formalised, as opposed to natural languages where only the syntax can). Naturally, the best way we have to ensure this is to use mathematics. There are three main approaches.

    1. Denotational semantics   denotational

      Each phrase of the language is interpreted as a denotation. Denotations often are mathematical objects inhabiting a mathematical space. These objects represent the effect of the execution, so the focus is on what and not on how.

    2. Operational semantics   operational

      Each phrase of the language is specified by the computation it induces. This definition is the most similar to my informal intuition, except that it still involves a fair amount of abstraction (for instance, an abstract machine could be used to specify the meaning of phrases). The focus is on how and not on what. Lambda calculus is probably the first example of operational semantics, used to define the semantics of Lisp. It is also an example of a semantics that uses synctactic transformations on phrases of the language itself.

    3. Axiomatic semantics   axiomatic

      The meaning of phrases is given by the logical axioms that apply to them (before and after). It ignores many details of computation and just concentrate on "invariants". It is probably the most abstract definiton, but also the one that better captures the underlying meaning of a programming language. An example is the following (the logical axioms are in curly brackets): 

      {a > 0} x=a; y=0; while(1<x){x=x/2; ++y;} {y=floor(log(a))}
  12. Source-to-source compiler   sourcetosource transpiler

    Also known as transcompiler or transpiler, it is a type of compiler which translates source code written in one programming language into source code in another programming language. It translates between programming languages that operate at approximately the same level of abstraction, while a traditional compiler translates from a higher level programming language to a lower level programming language. The process of translation is called transpiling.

  13. Statement   statement

    It is the smallest standalone element of an imperative programming language that expresses some action to be carried out. It is an instruction written in a high-level language that commands the computer to perform a specified action.

    1. Compound statement   compound

      In some languages (such as C, C++, Java, …), compound statements, also called blocks, are brace-enclosed sequences of statements.

  14. Type   type

    A type, also known as data type, is a classification of data which tells the compiler or interpreter how the program should use the data. It restricts the values that can be used in an expression (variable, function, …). It defines the operations that can be done on the data, the meaning of the data, and the way values of that type can be stored. Informally, the purpose of types is to make sure that variables and functions are always used in the way that they were intended to be used. Types can be seen as a (weak) specification language which is:

    • lightweight (few programmers would use it otherwise),
    • machine-checked (with minimal assistance),
    • ubiquitous (so it cannot be avoided!).
    1. Strong and Weak Typing   strong weak

      A strongly typed language is more likely to generate an error if the argument passed to a function does not closely match the expected type. This is not a formal distinction.

1.13 Security   security

1.13.1 Hashing

  1. Salt

    Salt should be at still as long as the final hashed output (256 characters if SHA-256 is used, for instance). It must be unique for every hashed string. It is safe to store it in the same table (it still makes bulk discover of passwords impossible, as equal passwords will have different hashes), but would be even better if put on a separate DB. Usual hashing functions (SHA-XXX and so on) are not safe to hash passwords in order to store them! See https://crackstation.net/hashing-security.htm for more info.

  2. Peanut

    A fixed salt to add to the random salt before hashing. This must not be stored on the DB, but in a property file or something similar.

1.13.2 Indentity and access management (IAM)   iam

Also known as identity management, it is the security and business discipline that enables the right individuals to access the right resources at the right times and for the right reasons.

1.13.3 Principal

Proper term to denote any entity that may be authenticated in a system (may be a human user, a service or anything else).

1.14 Terminology

1.14.1 Backlog   backlog

Generally refers to an accumulation of works waiting to be done or orders to be fulfilled.

1.14.2 Bootstrapping   bootstrap

The term comes from the expression "pulling yourself up by your own bootstraps" (where bootstraps are sort of strings that help to wear a boot). In computing, the original use is related to the bootstrap loader (see MBR), which is the first piece of code that runs when a machine starts, and is responsible to load the rest of the system. More in general, it now indicates any simple process which starts a more complex one. Bootstrapping is often seen in web frameworks.

1.14.3 Business process management (BPM)   bpm

Field in operations management that focuses on improving corporate performance by managing and optimizing a company's business. In particular, BPM software is a generic expression to describe software that is designed to ease or automate business processes.

1.14.4 Content management (CM)   cm

Set of processes and technologies that supports the collection, managing and publishing of information in any form or medium. In large settings, the process quickly becomes complex enough to require specific tools to ease or automate it.

1.14.5 Continuous delivery   continuousdelivery

It is an extension of continuous integration in which the (production) release process is automated too, and releases are done frequently (even daily or weekly).

1.14.6 Continuous deployment   continuousdeployment

It is an extension of continuos delivery in which every change that passes all stages of the production pipeline (build, unit tests, integration tests and so on) is immediately released to customers. There is no human intervention that triggers a release.

1.14.7 Continuous integration   continuousintegration

The practice of merging the code to a shared mainline several times a day. The main rationale is that this is done to avoid the integration hell. This practice should be coupled with version control, automatic builds and unit testing. In such a way, it is easy to immediately know which change introduced a problem.

1.14.8 Difference between executable and object file   executable object

An executable file is a complete program that can be run directly by an operating system (in conjunction with shared libraries and system calls). It generally contains a table of contents, a number of code blocks and data blocks, ancillary data (such as the memory addresses at which different blocks should be loaded), which shared libraries are needed, the entry point address and, sometimes, a symbol table for debugging. An object file has a lot in common, but it contains incomplete references to subroutines outside of itself, so that many machine instructions have only placeholder addresses. Before it is possible to run it, a linker must perform a final phase of compilation, resolving all the references and composing the final memory layout.

1.14.9 Infrastructure as Code (IaC)   iac

The process of managing and provisioning computer data centers through machine-readable definition files, rather than physical hardware configuration or interactive configuration tools. Infrastructure refers both to bare-metal servers and virtual machines. Not only does this give the possibility of automating many tasks, but it also gives the possibility of applying usual coding's best practices, such as version control. Typical approaches are:

Approach Keyword Phylosophy Description
Declarative Functional what Focuses on what the target configuration should be
Imperative Procedural how Focuses on how the infrastructure is to be changed to meet requirements
Intelligen Environment aware why Focuses on why the configuration should be in a certain way to ensure compatibility of all components

Environment aware is the next generation of IaC. Note that IaC is strictly related to DevOps, since it enhances the sharing of responsibilities, focuses on automation and speeds the deployment process up.

1.14.10 Reproducible builds   reproduciblebuild

Set of software development practices that create a verifiable path from human readable source code to the binary code used by computers. Basically, the build should be made a deterministic process. In particular, this requires giving a way to reproduce the build environment and the build steps. The whole purpose is security.

1.14.11 Software widget   widget

A relatively simple and easy-to-use application or component which is intended as an auxiliary tool which does not monopolize the user's attention. Related to this, GUI widgets are reusable modular components used to build a user interface.

  1. Web widget

    The same concept, applied to web pages. Web widgets can be written in a variety of technologies (for instance, a portlet is a sort of widget too). Note that the content of widgets is not visible to search engines, unless they are fully rendered server-side, so there can be concerns relating search engine optimization (SEO).

1.14.12 Standalone   standalone

Generic expression to describe software that does not require additional software to work. What is meant by "additional software" varies according to the context (it could be a specific computer program, an operating system, a network connection…).

1.14.13 Toolchain   toolchain

A set of programming tools that are used to perform a complex software development task. In general, the term chain refers to the fact that these tools are executed consecutively, with the output, or resulting environment state, of each tool becoming the input, or starting environment state, for the next one. By metonymy, it can be used even for tools which are not used consecutively. An example is a basic build toolchain composed of a compiler, linker, libraries and a debugger.

1.15 Theory   theory

1.15.1 Abstract analysis pattern   pattern algorithm

Recurring heuristics that can help in solving problems.

  1. Case analysis

    Split the input/execution into a number of cases and solve each case in isolation.

  2. Small examples

    Find a solution to small concrete instances of the problem and then build a solution that can be generalized to arbitrary instances.

  3. Iterative refinement

    Most problems can be solved using a brute-force approach. Find such a solution and improve upon it.

  4. Reduction

    Use a well-known solution to some other problem as a subroutine.

  5. Graph modeling

    Describe the problem using a graph and solve it using an existing algorithm.

  6. Write an equation

    Express relationships in the problem in the form of equations (or inequalities).

  7. Variation

    Solve a slightly different (possibly more general) problem and map its solution to the given problem.

  8. Invariants

    Find a function of the state of the given system that remains constant in the presence of (possibly restricted) updates to the state. Use this function to design an algorithm, prove correctness, or show an impossibility result.

1.15.2 Abstract data type (ADT)   adt data type

Mathematical model for data types which describes the semantics of the data type from the point of view of a user of the data, as opposed to data structures, which are concrete representations of data from the point of view of an implementer. In other words, an abstract data type corresponds to an interface and a data structure to an implementing class.

  1. Binary tree   binarytree

    A binary tree as in the graph theory sense. It is useful for representing hierarchy and is the building block of many useful abstract data types. Formally, it is a finite set of nodes which is either empty or consists of a root node together with two disjoints subsets which are themselves binary trees (right and left). It is said to be full if every node has exactly two children and perfect if it is full and all leaves have the same depth. It is balanced if for each node the difference in height for left and right subtrees is at most one. It is complete if every level is complete except the last, and all nodes on the last level are as far left as possible.

    1. Traversing   traversing visiting walking morris

      This is a key computation in binary trees. Visiting means reading data: pointers to nodes can move arbitrarily during the process. All visits take O(n) time and O(h) space, unless a parent pointer is maintained for every node. Inorder is O(1) space even without parent pointer (Morris traversal), but the algorithm is not thread-safe because it requires temporarily mutating the tree.

      1. Inorder   inorder

        First left, then root, then right.

      2. Preorder   preorder

        First root, then left, then right.

      3. Postorder   postorder

        First left, then right, then root.

  2. Binary search tree (BST)   bst binarytree

    A binary tree that allows for fast lookup, addition and removal of items. BSTs keep their keys in sorted order, that is, the key in each node must be greater than or equal to any key stored in the left sub-tree, and less than or equal to any key stored in the right sub-tree (binary search tree property). Leaves do not contain data. Note that if the tree is not balanced enough, operations become slow (on average, after random inserts and removals, O(\sqrt n)). In fact, running time of lookup, insertion and deletion is proportional to the height of the tree (O(\log n) in the best case).

    1. Self-balancing binary search tree   self balancing

      A binary search tree that automatically keeps its height small in the face of arbitrary item insertions and deletions.

      1. Red-black tree   red black

        A self-balancing binary search tree that uses a coloring of the nodes to ensure that the path from the root to the farthest leaf is no more than twice as long as the path from the root to the nearest leaf.

        1. Requirements
          • Each node is either red or black.
          • The root is black.
          • All leaves are black.
          • If a node is red, both its children are black.
          • Every path from a given node to any leaf contains the same number of black nodes.
      2. Treap   treap

        A binary search tree that uses a randomized approach to self balancing. Every key has a priority associated and priorities must satisfy the heap property. With high probability, the height of the tree will be O(\log n).

  3. Hash table   hashtable

    Implementation of an associative array abstract data structure (a structure that associates keys to values). A hash table uses a hash function that maps keys to buckets. Collisions are usually solved using singly-linked lists. In case the number of elements is known in advance, perfect hash functions can be used, which ensure constant running time.

  4. Priority queue   priorityqueue

    Same as a queue, but each element has a priority associated and dequeue returns the element with the highest priority. This abstract data type is usually implemented using a heap.

    1. Heap   heap binary tree

      A complete binary tree, whose purpose is to efficiently compute the maximum element (or minimum, if it is a min-heap). Insertion is O(\log n), lookup on max element is O(1) and deletion of it is O(\log n). It can be implemented using an array. The tree must satisfy the heap property, that is the value of a node P must be greater (smaller if it is a min-heap) of the values of its children. A BST can do anything a heap can do with same or better space complexity, but with possibly some implementation overhead.

    2. Fibonacci heap   fibonacci heap

      An implementation of a priority queue which uses a collection of heap-ordered trees. Insert, Find-min, Decrease-key and Merge have constant amortized running time, whereas Delete-min has O(\log n) complexity. The main idea is that operations are performed lazily, often adding new trees, and then from time to time heavy operations reestablish the balance. Fibonacci heaps are not considered very efficient in practice.

  5. List

    Abstract data type that serves as an ordered collection of elements. There are singly-linked lists, where each element has a pointer to the following one, and doubly-linked lists, where each element has a pointer to the preceding and the following ones.

  6. Queue   queue fifo

    Abstract data type that serves as a collection of elements with two principal operations: enqueue (to add an element) and dequeue (to remove the least recently added). It is also called FIFO (first-in first-out). It can be implemented as a singly linked list with a pointer to the top element (O(1) time complexity for enqueue and dequeue) or as an array (still O(1), but amortized because of resizing).

    1. Deque   deque

      Also called double-ended queue, it is the same as a queue but supports adding and removing from either side of the list. It can be implemented with a doubly linked list.

  7. Skip list   skiplist

    Data structure that allows fast search within an ordered sequence of elements. It uses a hierarchy of sorted and increasingly smaller linked lists (each one a subsequence of the collection). It implements the same functionality as balanced BSTs, but is easier to code and often faster, especially in a concurrent context.

  8. Stack   stack lifo

    Abstract data type that serves as a collection of elements with two principal operations: push (to add an element) and pop (to remove the most recently added). It is also called LIFO (last-in first-out). It can be implemented as a singly linked list with a pointer to the top element (O(1) time complexity for push and pop) or as an array (still O(1), but amortized because of resizing).

1.15.3 Design pattern   pattern algorithm

There are some recurring design patterns for algorithms too.

  1. Sorting

    Sorting is often a good starting point, but only when good solutions have at least O(n\log n) time complexity.

  2. Recursion

    Appropriate when the solution can be progressively reduced to easier cases. A classical example is string matching.

  3. Divide and conquer

    Divide the problem into smaller problems, solve them independently and then combine the results. It is not the correct approach when a solution heavily depends on global properties of the input. It is similar to recursion to some extent, but more specialized (in recursion subproblems do not have to be of the same form).

  4. Dynamic programming (DP)

    Applicable when a problem has the optimal substructure, which means that it is possible to reconstruct a solution from solutions to subinstances. This pattern can often be used to speed-up recursive solutions, at the cost of increased space usage.

  5. Greedy method

    Make locally optimum decisions and do not change them. Most of the times this only produces approximate solutions, but it can be a good starting point.

  6. Incremental improvement

    Sometimes an optimal solution can be found by refining suboptimal ones. Examples are bipartite matching, maximum flow, computing all pairs of shortest paths in a graph. It is often useful when designing heuristics.

  7. Elimination

    Finding a way of recognizing and excluding suboptimal solutions is a good way of speeding an algorithm up. Binary search is an example of elimination.

  8. Parallelism

    This is a good practical way of speeding up an algorithm whose solution depends on the solutions of multiple independent instances. An example is sorting a huge array of numbers.

  9. Caching

    Useful in multiple situations, use when an expensive operation is repeated multiple times.

  10. Randomization

    Useful in multiple situations, often to design heuristics that work well in practice. An example is finding the median of an unsorted array.

  11. Approximation

    Mainly used to solve NP-complete problems.

  12. State

    Formally, the state of a system is information that is sufficient to determine how that system evolves as a function of future inputs. Identifying a good notion of state for a problem gives the possibility to concentrate on what is important and to eliminate the "noise".

1.15.4 Formal language   formal

Given an alphabet Σ, a formal language is a subset of Σ*. Elements of a formal language are called words. Formal languages are used in many situations: for instance, to define the syntax of programming languages, to specify decision problems (note that complexity classes are defined as sets of formal languages that can be parsed by machines with limited computational power), or to represent the syntax of axiomatic systems in logic.

  1. Regular language   regular

    The collection of regular languages over an alphabet Σ is defined recursively as follows:

    • The empty language and the empty string language {ε}.
    • For each a∈Σ, the singleton language {a}.
    • If A and B are regular languages, then \(A\cup B\) (union), \(A\bullet B\) (concatenation), and \(A^*\) (Kleene star) are regular languages.

    In simpler words, a regular language is a formal language that can be parsed by an automaton with a constant amount of memory and only one pass. Note that regular expressions originate from this notion (Kleene's theorem actually states the equivalence between regular expressions and finite automatons).

    1. Equivalent definitions

      There are many equivalent definitions of a regular language. Among the most interesting:

      • it can be accepted by a read-only Turing machine;
      • it can be defined in monadic second-order logic;
      • it is recognized by some finite monoid M, meaning it is the preimage \(\{w\in\Sigma^*|f(w)\in S\}\) of a subset S of M under a monoid homomorphism \(f:\Sigma^*\to M\).
